            Access tables — minimal viable model
            tblCustomer
            • CustomerId — AutoNumber, PK
            • Name — Text(120)
            • LogoPath — Text(255)
            • PrimaryContact — Text(120)
            • Notes — Memo
            tblTenant
            • TenantId — AutoNumber, PK
            • CustomerId — FK → tblCustomer
            • TenantDomain — Text(120)
            • TenantGuid — Text(40)
            • Environment — Number (Prod/Dev/Test)
            tblAudit
            • AuditId — AutoNumber, PK
            • CustomerId — FK → tblCustomer
            • TenantId — FK → tblTenant
            • CatalogVersionId — FK → tblCatalogVersion
            • PreviousAuditId — FK → tblAudit, nullable
            • StartedAt / CompletedAt
            • Status — Number
            • Reviewer — Text(120)
            tblCatalogVersion
            • CatalogVersionId — AutoNumber, PK
            • Version — Text(20) e.g. "2026.04"
            • ImportedAt — Date/Time
            • SourceFile — Text(255)
            • ItemCount — Number
            tblCheckCategory
            • CategoryId — AutoNumber, PK
            • Name — Text(120), unique
            • ProductDomain — Text(40)
            • SortOrder — Number
            tblCheckItem
            • CheckItemId — AutoNumber, PK
            • CategoryId — FK → tblCheckCategory
            • CatalogVersionId — FK → tblCatalogVersion
            • ShortDescription — Memo
            • ExpectedValue, HowToRead — Memo
            • PermissionsNeeded — Text(255)
            tblFrameworkMapping
            • MappingId — AutoNumber, PK
            • CheckItemId — FK → tblCheckItem
            • Framework — Number (ISO/CIS/NIST)
            • ControlId — Text(40) e.g. "A.8.5"
            tblFinding
            • FindingId — AutoNumber, PK
            • AuditId — FK → tblAudit
            • CheckItemId — FK → tblCheckItem
            • Status — Number (Pending/Pass/Fail/NA)

            • CheckedAt — Date/Time
            tblComment
            • CommentId — AutoNumber, PK
            • FindingId — FK → tblFinding

            • EvidenceId — AutoNumber, PK

            • LogId — AutoNumber, PK

            Open questions

            Confirm before build
            • Single-DB or per-customer DB? Recommend per-customer .accdb in a customer folder — cleaner isolation, easier handover, no row-level filtering.
            • UI framework: WPF vs WinForms vs WinUI 3. Recommend WPF for richer data binding and a more polished look at no extra cost.
            • Graph queries: embed Microsoft.Graph .NET SDK, or shell out to PowerShell scripts? Recommend hybrid — SDK for simple reads, PowerShell for the existing complex commands in the catalog (Get-MgServicePrincipalAppRoleAssignment, etc).
            • Multi-user: Access does not love simultaneous writers. If two reviewers will share an audit, plan to split data (back-end .accdb on a share with linked tables) or move to SQLite/SQL Server later.
            • Evidence files: store paths in DB, files on disk in <customer>\evidence\<findingId>\. Hash on import.

            Langate · Executive summary Acme Holdings AB

            Headline

            Acme’s tenant scores well on identity hygiene (MFA, Conditional Access baseline) but has elevated risk in three areas: legacy authentication, broad service-principal permissions, and missing Defender for Cloud Apps governance.

            Risk profile

            High-risk findings12require remediation within 30 days
            Medium-risk findings19plan within 90 days
            Low-risk findings7track in backlog

            Top recommendations

            1. Block legacy authentication tenant-wide via Conditional Access.
            2. Inventory and review service principals with Directory.ReadWrite.All; remove or scope down.
            3. Enable Defender for Cloud Apps OAuth governance and review unsanctioned third-party apps.
