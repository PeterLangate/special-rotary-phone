Chapter 1 · Introduction

Chapter 1What SecAudit is

SecAudit is a structured workflow tool for performing a Microsoft 365 and Azure security audit against a customer tenant. It walks an auditor through a 624-item catalog, captures Pass / Fail / N/A findings with risk severity, supports remediation planning across audits, and produces deliverables for the customer.

Who uses it

SecAudit is built around two roles. The auditor at Langate runs the audit, captures findings, attaches evidence and signs off accepted risks. The customer admin can be invited to fill in the customer-interview section (process and policy items that cannot be answered from Microsoft Graph alone) but otherwise reads only the deliverables.

The model in one paragraph

An audit belongs to one customer and targets one or more tenants. The audit is anchored to a catalog version (e.g. 2026.04) which fixes the set of 624 check items to evaluate. For each item the auditor records a finding: a status, a risk severity, an observation, optional remediation, attached evidence, and — on failures — a remediation lifecycle (open → acknowledged → remediating → verified, or accepted with sign-off). Findings carry an owner and a target date. All edits are local, instant, and undoable with Ctrl+Z.

Two views of the same data

The workspace view

Per-item editing surface. Three panes: categories on the left, items in the middle, finding detail on the right. Every keystroke saves.

The status-report view

Aggregated, customer-facing view. Risk donut, severity-tinted heatmap, KPIs, findings grouped by category. Printable to PDF.

Document map

Part I — Chapters 2–11 walk through every screen and feature of the tool, in the order an auditor would encounter them. Part II — Chapters 12–14 outline two ways to ship SecAudit: a local C# WPF desktop app, or a tenant-hosted SPFx web application. Chapter 14 ends with a side-by-side decision matrix.

Chapter 3 · Audit workspace

Chapter 3Audit workspace

The workspace is where 90% of an audit is performed. Three vertical panes: categories on the left, items in the middle, finding detail on the right. The middle and right panes update as you click.

3.1 The three panes

Categories pane (left)

Lists the 61 catalog categories, each with a per-category progress fill that shows how many items have been answered. The active category is highlighted in primary blue with a green left edge. A search box at the top filters categories in place.

Items pane (middle)

Lists the items in the active category. Each row carries:

  • A selection check-box for bulk operations
  • A status icon (Pending, Pass, Fail, N/A)
  • A product-domain icon (Entra, Defender, Purview, etc.)
  • The item description and ID
  • Status / risk badges and lifecycle / overdue / stale pills
  • Per-row PowerShell and Portal action buttons

Three filter rows sit above the list. The status row (All / Pending / Pass / Fail / N/A / Stale / Lifecycle), the risk row (All / High / Med / Low / Overdue / Mine), and the tenant row (All / Prod / Dev). All three filters compose — an item is shown only if it passes every active filter.

Detail pane (right)

Shows the active item: its expected value, “how to read” instructions, permission needed, and the editable finding (status, risk, observation, remediation, evidence). For failed items it also shows the remediation lifecycle stepper, framework mapping (ISO/CIS/NIST), and the comment thread. Two AI-assist buttons offer to draft remediation steps or rewrite the observation in customer-friendly language.

Chapter 3 · Risk indicator

3.2 The visible risk indicator

Every Fail item carries a 4-px colored left bar and a tinted Fail badge. This is the single most important visual cue in the workspace — severity is decodable from a 1500-pixel-wide list at a glance.

Item list — visual cue
#13 · Phishing-resistant MFA available Fail · HIGH
#14 · SMS / Voice MFA still enabled Fail · MED
#110 · PIM-eligible roles still permanent Fail · LOW
#15 · Number matching enabled Pass

When the same row is also the active selection, the risk color wins over the primary-blue active indicator. This guarantees that severity stays visible while you edit.

3.3 Setting status and risk

The detail pane offers a four-button status row (Pending / Pass / Fail / N/A) and a three-pill risk row (Low / Med / High). The two work together:

ActionEffect
Click PassSets status=pass, clears risk to none, removes any lifecycle.
Click FailSets status=fail. If no risk yet, risk defaults to med. Pre-existing risk is preserved.
Click Med on an active itemSets risk=med. Item must already be a fail (the tool warns “Set status before risk” otherwise).
Click N/ASets status=na, clears risk and lifecycle. The item is excluded from the risk donut.

All four also have keyboard shortcuts 1 Pending, 2 Pass, 3 Fail, 4 N/A. The risk pills have L / M / H. See chapter 5 for the full list.

Why Fail defaults to Med, not None
Marking an item as failed without a severity hides it from the risk donut and the heatmap severity tint. To avoid silent disappearances, the tool defaults a fresh failure to medium risk and lets you override with H or L immediately.

Chapter 4 · Finding lifecycle

Chapter 4Finding lifecycle & risk acceptance

Failed items don’t just sit on a list — each one moves through a documented remediation lifecycle, or is formally accepted as a known risk with a sign-off. Both paths produce evidence the customer can take to a compliance review.

4.1 The four-step remediation track

Every failed item starts as Open. Click any step in the lifecycle stepper on the detail pane to advance the state. The stepper appears only on failures — pass items have no lifecycle.

Open
Acknowledged
Remediating
Verified
StateMeaningReveals fields
OpenDefault. Captured but no plan yet.—
AcknowledgedCustomer is aware. Not yet planned.—
RemediatingActively being fixed. Has owner + due date.Owner, target date
VerifiedFix applied. Re-run the catalog check to confirm; the finding will move to Pass.—

4.2 Risk acceptance — the side track

Some findings cannot be fixed because the customer has compensating controls or an explicit business decision to bear the risk. These are accepted, not remediated. Click the Accept risk button (lock icon, far right of the stepper) to move the finding off the remediation track.

On first acceptance the tool pre-fills Signed by (the active reviewer), Signed at (today), and Expires (one year out). The auditor edits these and writes a justification — what the compensating control is, why the risk is acceptable.

Acceptance expiry banner
When today is within 30 days of Expires, an amber banner appears in the detail pane prompting a renewal review. After expiry the banner turns red and asks the auditor to re-validate or move the finding back to Open.

4.3 Bulk-assigning owner and target date

From the bulk bar (chapter 5) the Assign… button opens a small popover with two fields: owner email and target date. Apply once and every selected fail item gets an owner, a target, and a lifecycle of remediating. Items not in fail state are skipped silently.

4.4 Overdue and due-soon visual cues

On every fail item that has a target date in remediating, the row shows one of:

Overdue  Pulses red. Target date is in the past.
Due soon  Static amber. Target date is within 14 days.
No badge  Target date is more than 14 days out, or item is not in remediating.

The risk filter chip Overdue (chapter 6) restricts the item list to overdue items only — the fastest way to triage SLA breaches mid-audit.

Chapter 5 · Bulk & keyboard

Chapter 5Bulk operations & keyboard

Audits are won and lost on speed of capture. SecAudit is built around two patterns that turn an 8-hour audit into a 2-hour audit: bulk multi-edit, and keyboard-only navigation backed by an undo log.

5.1 Selecting multiple items

Click the check-box on the left of any item, or press X on the active item to toggle its selection. The bulk bar slides in once at least one item is selected and shows a running count. Click Select all visible to tick every row currently visible (filters apply); click Clear to empty the selection.

Switching to a different category clears the selection automatically and a footer note confirms it. This avoids the silent-selection trap where a bulk action applies across categories.

Chapter 5 · Command palette

5.4 The command palette — Ctrl+K

The command palette is the fastest way to navigate 624 items. Press Ctrl+K from anywhere to open a search modal that filters items by description and category in real time.

Search forms

  • Free text: legacy → matches every item whose description or category contains "legacy".
  • Direct ID: #17 or 17 → jumps straight to item 17.
  • Category jump: conditional → offers both items and a "jump to Conditional Access" row.
  • Empty search: shows current failures first — useful for resuming yesterday’s work.

Navigation in the palette

↑ / ↓
move highlight
↵
jump to selected result — opens workspace, selects category, activates item
Esc
close palette
Click anywhere outside

Why this exists
With 624 items across 61 categories, scrolling for a known check by name is the slowest path. The palette resolves the lookup in under a second and the jump opens the right category and selects the item, saving two clicks.

5.5 The undo log

Every state change — single edits, bulk operations, owner assignments, lifecycle transitions, evidence attachments — is pushed to an in-memory undo stack. Ctrl+Z pops the top entry and restores the previous snapshot. There is no redo at present; an accidental undo can be reverted by re-applying the action.

The undo log lives only for the current session. On close it is discarded; the persisted database / list keeps only the latest state. This is intentional: the audit trail of who changed what when is captured separately in the activity log (chapter 9 of the deliverables view).

6.2 The tenant posture heatmap

On the dashboard, every catalog item is rendered as one cell in a 624-cell grid. Cells are color-coded by status and (for fails) by severity. Hover for a tooltip, click to jump to the item in the workspace.

Cell colorMeaning
GreenPass
RedFail · High risk
AmberFail · Medium risk
TealFail · Low risk
GrayN/A
Light grayPending
Amber corner markStale evidence (verified > 90 days ago)
Blue corner markIn remediation lifecycle

The heatmap uses the same severity palette as the row indicator and the audit report findings list. Once a customer has the language — "the red cells in the lower right are our high-risk Conditional Access items" — it works unchanged across every screen.

6.3 The per-tenant rollup

Above the heatmap on the dashboard, the rollup shows one card per tenant in the audit. Each card carries a stacked progress bar and four counts: pass / fail / n·a / pending. This is the surface to look at first when the audit covers multiple tenants — it answers “is the gap between Prod and Dev growing?”.

6.4 Color-blind safe mode

Toggle the CB button in the title bar (or write a settings rule) to swap every red / amber / teal surface for one decodable without color:

  • Risk-tinted left bars become diagonal hatch patterns — high uses 45° red-on-white, med uses 0° amber-on-white, low uses −45° teal-on-white. Different angles per severity.
  • Fail badges prepend a glyph: !! for high, ! for med, · for low. The glyphs are decodable in monochrome.
  • Heatmap cells get the same hatched fills.

CB mode is per-session. Storing it in customer-facing exports is on the roadmap but not yet present.

6.5 The reference catalog

Switch to the Reference catalog screen to browse the underlying 624 items without recording a finding. The category filter on the left and the search box at the top both narrow the table. Use this to look up the “how to read” instructions for an item without leaving the read-only view.

Chapter 7 · Evidence, comments, AI

Chapter 7Evidence, comments, AI assist

A finding without backing material is contestable. SecAudit attaches three layers of supporting context to every check: structured evidence files, free-form comments, and AI-drafted remediation / customer translations.

7.1 The evidence dropzone

On the detail pane, below the observation and remediation textareas, the evidence section accepts files in two ways:

  • Click the dropzone — opens a file picker (in the local C# build) or a URL prompt (in the SPFx build).
  • Drag a file from the OS — the dropzone highlights green; release to attach.

Attached files render as thumbnail tiles with a type icon. PNG / JPG / SVG show a picture icon, CSV / XLSX show a clipboard icon, JSON / TXT / LOG / EML show a document icon. Hover a tile and click the small × button to remove. Removal is undoable with Ctrl+Z.

Evidence storage
In the C# build, files are stored on disk under <customer>\evidence\<findingId>\ with SHA-256 hashes. In the SPFx build, files go into a SharePoint document library under the audit’s site. Both record only the relative path in the database.

7.2 Comments — collaboration thread per finding

The comments block under each finding is a chronological thread. The auditor and (in the SPFx build) the customer admin can both post. Use comments for:

  • Conversation with the customer admin about a specific finding
  • Internal notes for the QA reviewer
  • Linking out to a Jira / Planner ticket for the remediation

Comments are not part of the customer deliverable by default — they live on the workspace and the internal QA report. The full technical findings deliverable offers an opt-in flag to include them.

7.3 AI assist — Draft remediation / Translate for customer

Two small sparkle-icon buttons sit next to the Observation and Remediation textareas:

ButtonWhat it drafts
Draft remediation (next to Remediation)Step-by-step instructions tailored to the item’s product domain (Entra, Defender, Purview, Azure, etc.) including portal location, recommended setting, verification command, estimated effort, and suggested owner team.
Translate for customer (next to Observation)Rewrites the technical observation into plain language for the customer-facing deliverable. Keeps the meaning, drops the jargon.

Both buttons render an inline preview with three actions: Apply replaces the textarea content, Regenerate drafts a new variant, and Dismiss closes the preview. The auditor always reviews before applying — AI output is a starting point, not a deliverable.

Chapter 7 · PowerShell & Portal

7.4 Per-item PowerShell & Portal buttons

Two small buttons live on every item row and again on the detail pane. They open a focused single-item action; P and O from the keyboard do the same.

ButtonBehavior
PowerShellOpens a modal showing the catalog command for this item, syntax-highlighted. Copy to clipboard, or (in the local C# build) press Run to execute against the connected tenant.
PortalOpens the relevant Microsoft 365 / Azure admin portal page in a new browser tab — e.g. Entra Conditional Access for items in that category, Defender XDR for Defender items.
Why both
Some items are answered fastest by running a Graph cmdlet (and the output is the evidence). Others are answered by clicking through the admin UI to confirm a policy. Both buttons are always offered — the auditor picks the path that fits this customer.

Chapter 8 · Reports & comparison

Chapter 8Reports & year-over-year comparison

SecAudit ships with two interactive views (the live audit status report and the comparison view) plus six pre-built deliverables on the Reports screen. All are generated from the same finding data — the auditor never enters numbers twice.

8.1 The live audit status report

From any screen press R or click Audit status report to open a modal that summarises the engagement. The structure:

  1. Cover — customer name, tenant, reviewer, started date, status badge, and four risk tiles (High / Med / Low / Accepted) showing the live counts.
  2. Overall progress — coverage %, item-count KPIs, a stacked bar (pass / fail / n·a / pending), and a risk donut showing the failure mix.
  3. Tenant posture heatmap — the same severity-tinted grid as the dashboard, embedded in the report.
  4. Findings by category — one row per category with item counts, progress %, severity mini-bar (high / med / low mix), 4-audit trend sparkline, and benchmark position vs. the portfolio.
  5. Recorded findings — every finding grouped by category, with risk-tinted left border, status badge, observation, evidence list, and framework mapping (ISO / CIS / NIST). Filter chips at the top: All / Failures only / High risk / Medium risk / Low risk / Accepted / Passes / N/A.
  6. Pending items — what is still outstanding, capped at 60 with a count of any further hidden items.

Click Print in the modal footer to send the report straight to a print-to-PDF dialog. The print stylesheet hides all chrome (sidebar, title bar, modal footer, action buttons) and prints just the report contents at A4.

For customers on their second or later audit, the workspace header shows a Compare with previous button. Clicking it opens a modal that diffs the current findings against the previous audit’s findings, by CheckItemId:

The comparison is the basis of the customer renewal conversation: how much improved, where regressions appeared, what new compliance items the catalog now requires.

8.4 The activity feed

On the dashboard, the Activity panel shows the last 24 hours: comments posted, evidence attached, status changes, PowerShell commands run, catalog re-imports. Click any row that references a specific item to jump straight to the workspace with that item active.

The activity feed is the audit trail used by the internal QA review and is preserved in the database independently of the per-finding undo log.

Chapter 9 · Customer interview

Chapter 9The customer interview

About 45 of the 624 catalog items cannot be answered from Microsoft Graph alone — they are process, policy, and training questions that require a human inside the customer organisation to answer. SecAudit groups them on a dedicated screen that the auditor can run through during a live call or share with the customer for self-service.

9.1 What goes on the interview screen

The Customer interview screen pulls in every catalog item flagged as interview-only. Each question carries:

  • The catalog ID (e.g. #101)
  • The question phrased as a question (the catalog descriptions are auto-rephrased — "Phishing simulation programme" becomes "Is a phishing simulation programme in place?")
  • Four answer buttons: Yes, Partial, No, N/A
  • A free-text notes field for context the customer wants on the record
  • An “answered by / when” line that records who provided the answer

Questions are grouped by category so the auditor can move through related items in one pass. A progress bar at the top shows percent answered out of the total.

9.2 Live-call mode

During a customer call, the auditor opens the interview screen on a shared screen and walks through the questions. Click an answer button, type a short note, move on. Each click writes to the in-memory model immediately; on save, the answer becomes the finding for that catalog item.

Estimated pace: ~45 questions in under an hour.

9.3 Self-service share link

Click Share with customer in the screen actions. The tool generates a one-time URL such as https://secaudit.langate.se/interview/acme/<token> and copies it to the clipboard. The customer admin opens the URL in any browser and answers at their own pace; their answers stream into the same backing store.

Build-mode difference
In the C# build, the share link is hosted by a small ASP.NET endpoint that the local app spins up on demand and tunnels via a reverse proxy. In the SPFx build it is just a tenant URL that the customer admin (already authenticated to the tenant) can hit directly. The SPFx flow is materially simpler.

9.4 Outstanding-questions badge

The Customer interview entry in the left navigation rail carries a small numeric badge that counts unanswered questions. It only shows when there is at least one outstanding question, and disappears once every interview item has an answer. Use it as a "ready to deliver" signal — if the badge is gone and the workspace coverage is 100%, the audit is ready for the QA review.

Chapter 10 · Benchmarks & trends

Chapter 10Cross-customer benchmarks & trends

A finding lands harder when the customer can see how their tenant compares to the rest of Langate’s portfolio — and how it has moved since last year. SecAudit aggregates anonymised pass-rates across delivered audits and renders them per category.

10.1 The portfolio benchmark widget

On the dashboard, the “Portfolio benchmarks — areas to focus” panel shows the eight categories where this customer is furthest below the portfolio median. Each row carries:

  • Category name with product icon
  • This customer’s pass-rate as a percentage
  • Benchmark bar — a 100mm horizontal scale showing the portfolio Q1, median, Q3, and top quartile, with this customer’s position marked as a colored dot
  • Delta — difference between this customer and the portfolio median, in percentage points
  • Cohort size — how many delivered audits the benchmark draws from

The dot colour matches the band the customer falls into: green if top quartile, light green above median, teal at median, amber below median, red bottom quartile.

10.2 In the audit status report

The same benchmark bar appears in the “vs portfolio” column of the findings-by-category table inside the status report, alongside a 4-audit trend sparkline. Together they answer two customer-facing questions in two columns:

Where do we stand vs the portfolio?

The benchmark bar position. A teal-or-greener dot is good; an amber or red dot calls for a remediation plan focused on that category.

Are we improving over time?

The 4-audit sparkline. A rising line in green = improvement; a falling line in red = regression. Hover to see the percentages.

10.3 Stale-evidence tracking

Every finding records a checkedAt timestamp set automatically when the auditor records / re-records the finding. After 90 days the finding is flagged Stale. The marker shows up in three places:

  • An amber pill on the workspace item row
  • An amber corner mark on the heatmap cell
  • A “stale” line in the lifecycle metadata in the detail pane (with day count)

The Status filter Stale chip narrows the workspace to only the stale items — the obvious starting point for a re-verification pass before delivery.

10.4 Per-category trend sparklines

Each category that has been audited at least twice has a 4-audit pass-rate history (currently sourced from the customer’s own previous audits; extending to portfolio-trend is on the roadmap). Sparklines render in red if the trend is downward, green if upward.

Chapter 11Catalog versions & multi-tenant scoping

Two long-running concerns sit at the engagement boundary rather than inside a single finding: which version of the 624-item catalog this audit was performed against, and which tenant a finding actually targets when the customer has more than one.

11.1 The catalog-version banner

The workspace renders a status banner at the top of the screen on every visit. Two states:

StateBanner
On latest catalogGreen border. Reads "This audit is on the latest catalog 2026.04."
BehindAmber border. Reads "Audit on catalog 2026.01 · latest is 2026.04: 4 added · 3 changed · 1 removed." With a "View diff ›" link.

11.2 The catalog-diff modal

Click View diff › to open a modal listing every change between the audit’s catalog version and the latest. Three sections, each with a count summary tile:

  • Added — new items that did not exist in the audit’s catalog. Each entry shows the new item ID, description, and a "reason" line explaining why it was introduced (e.g. "M365 Copilot — new domain, 9 controls added").
  • Changed — items whose definition was tightened or expanded between versions (e.g. "Authentication strengths: phishing-resistant now baseline (was optional)").
  • Removed — items retired from the catalog, usually because they were superseded by a newer unified item.

The modal’s Promote audit to latest action carries the audit forward: any new items become pending, any changed items get a "review" flag, removed items are archived but their history is preserved. The auditor decides when to promote — mid-engagement promotion is supported but not forced.

Why a versioned catalog matters
Two engagements in the same calendar quarter can be on different catalog versions if Microsoft adds a new product domain (M365 Copilot, Defender for Cloud Apps governance) between them. Without versioning, every previous audit would silently appear "incomplete" against the new catalog. With versioning, each audit’s coverage is unambiguous.

Chapter 11 · Multi-tenant scoping

11.3 Multi-tenant scoping

Many customers have more than one Entra tenant — typically Production and Development, sometimes a separate Acquisition tenant. SecAudit stores tenants on the audit (tenants[]) and on each finding (TENANT_ASSIGNMENT[itemId]). The default tenant is the audit’s primary; deviations are recorded on the finding.

11.4 The tenant rollup on the dashboard

Above the heatmap, the dashboard shows one card per tenant in the audit. Each card carries a stacked progress bar (pass / fail / n·a / pending) and four counts. Use this to spot the gap between tenants — "Prod looks healthy at 91% pass, Dev is dragging at 64%" — before opening the workspace.

11.5 The tenant filter chip in the workspace

The third filter row in the workspace is the tenant scope. Three chips: All tenants, Prod, Dev. Setting any chip other than All hides items not assigned to that tenant from both the item list and the bulk-bar selection.

Filters compose: setting Status=Fail, Risk=High, Tenant=Prod gives "high-risk production failures only", which is the typical engagement-close conversation.

11.6 Per-tenant evidence and reports

Evidence files attached to a finding inherit the finding’s tenant scope implicitly. The audit-status-report filters fold in the tenant scope: filtering to Prod-only produces a Prod-only report. The full technical findings deliverable offers a per-tenant flag — one report per tenant, or a combined report with per-section dividers.

When to scope per-tenant
Always when the customer’s Dev tenant is Microsoft-managed sandbox data and not a real production target — the auditor still notes findings, but the deliverable focuses on Prod. Always when the customer requests separate sign-off lines for two business units.

The first option ships SecAudit as a single Windows desktop application, signed with a Langate code-signing certificate, distributed by MSIX. 12.2 The PowerShell button — the killer feature of this build

Every catalog item carries a one-line PowerShell command that produces the evidence (e.g. Get-MgServicePrincipal -Filter "appOwnerOrganizationId eq null"). In the C# build, the Run button in the per-item PowerShell modal executes that command directly:

  1. The user clicks Run in the PowerShell modal
  2. SecAudit lazily starts an in-process PowerShell 7 runspace if none is open
  3. The runspace inherits the user’s existing Connect-MgGraph token (or prompts if absent)
  4. The catalog command runs; stdout / stderr stream into the modal
  5. The output is automatically attached as evidence (output-{itemId}-{timestamp}.txt) to the active finding
  6. The finding’s checkedAt is set to now; status remains for the auditor to set

One-click verification turns a 30-second copy-paste-into-PowerShell-window-and-back into a 2-second flow. Across 624 items this adds up to roughly 4 working hours saved per audit.

Why in-process, not Start-Process
Spawning a new powershell.exe per command would lose the Connect-MgGraph token after every run, forcing the user to re-authenticate. An in-process runspace inherits the token; the user logs in once at app startup.

12.3 The Access database

Each customer gets its own .accdb file in <customer>\SecAudit.accdb. The schema is small — ten tables connected by foreign keys (covered in detail on the Architecture screen of the mockup). Per-customer files have three benefits:

  • Cleaner isolation — deleting a customer file deletes that customer’s audit history with no row-level filtering
  • Easier handover — the entire audit can be zipped and sent to the customer
  • Portable backup — just copy the file

Chapter 13Build option 2 — Tenant-hosted SPFx web app

The second option ships SecAudit as a SharePoint Framework (SPFx) application deployed into the Langate tenant (or, for very large engagements, into the customer’s tenant). All audit data lives in SharePoint lists. All checks run as REST queries against Microsoft Graph from the user’s browser. No local PowerShell, no .accdb files, no installation — but multi-user from day one.

13.1 Architecture at a glance

SecAudit.SPFx · Single-Page Application
UI layerSPFx web part · React 18 · TypeScript · Fluent UI 9 with Langate brand overrides
IdentityEntra ID via SPFx SSO. AadHttpClient for Graph. Tenant-wide consent.
Data layerSharePoint lists (Audits, Findings, Comments, Evidence, CatalogItems, Tenants, Activity). PnPjs for queries.
Microsoft GraphRead-only REST queries against /v1.0 and /beta endpoints. No PowerShell.
Evidence storageSharePoint document library, one folder per finding, native version history.
ReportsClient-side HTML → print-to-PDF. Server-side DOCX via a small Azure Function for pixel-perfect Word output.

13.2 No local PowerShell — the consequence

The SPFx build cannot start a PowerShell process. Instead, the catalog defines a Graph REST query for every check that has a one-line equivalent. The per-item button in the workspace becomes Run check:

  1. The user clicks Run check on the active item
  2. SPFx calls Microsoft Graph via AadHttpClient using the user’s SSO token
  3. The JSON response is rendered in the same modal that the C# build uses for stdout
  4. The JSON is auto-attached as evidence (graph-{itemId}-{timestamp}.json) and stored in the SharePoint document library
  5. The finding’s checkedAt is set to now; status remains for the auditor to set

About 480 of the 624 catalog items have a clean Graph REST equivalent. The remaining 144 are mostly Azure resource-management checks (AKS configuration, Storage, Networking) that require Azure ARM REST queries; these run via the same AadHttpClient against management.azure.com. A small residue of ~30 items have no REST equivalent today — they fall back to a "copy-paste this PowerShell into your shell" instruction with a copy button.

High tenant permissions
For SecAudit to read the customer tenant via Graph, the SPFx app needs tenant-wide admin consent for the union of catalog permissions (Directory.Read.All, Policy.Read.All, SecurityEvents.Read.All, AuditLog.Read.All, etc.). Acquired once per customer engagement by the customer’s Global Administrator and revocable at any time.

Chapter 13 · Multi-user model

13.4 Multi-user from day one

The SPFx build inherits the SharePoint co-authoring model. Three or four Langate auditors can edit the same audit at the same time:

  • Findings written by another reviewer appear in real time via the SharePoint change-feed (PnPjs useEffect + SharePoint webhook)
  • Comment threads are live; a customer admin invited as guest sees comments as the auditor types them
  • The activity feed shows everyone’s actions, not just the local user’s

For large engagements (one Langate lead + two specialists + one customer point-of-contact = 4 active editors) this is a category-of-magnitude improvement over the C# build’s "send the .accdb back and forth" model.

13.5 The customer interview — trivially shareable

Because SPFx runs in the customer tenant (or in a tenant the customer admin can be invited to as guest), the Share with customer link becomes a plain SharePoint URL. The customer admin opens it, signs in with their existing tenant credentials, and answers. No tunneling, no anonymous-access surface.

Chapter 14 · Recommendation

14.3 Recommendation: build both, in this order

Ship the C# WPF build first. It is the smaller cone of work (single-user, no SharePoint plumbing, no tenant deployment), it preserves the killer one-click PowerShell flow, and it covers ~80% of current Langate engagements which are single-auditor jobs. Target: 8–10 weeks of development time including reports, signing, and MSIX distribution.

Ship the SPFx build next, sharing the same audit data model (the Access tables map directly to SharePoint lists), the same brand theme, and the same finding lifecycle. Catalog REST queries are added incrementally — the SPFx build can launch with a 90% catalog coverage and gain the rest over the following two release cycles. Target: 12–14 weeks once the C# build is in production and the data model is settled.

Both builds share this documentation and the mockup as the source of truth for behaviour. A user manual change made in chapter 5 is equally true for both. The catalog is one source — the same Excel file imported into both Access (C#) and SharePoint Lists (SPFx).

One catalog, two front-ends
The 624-item Excel catalog is the contract between the two builds. As long as both builds import the same version, both produce the same finding model, the same report structure, and the same year-over-year comparison. A customer that starts on the C# build can move to SPFx mid-engagement (or vice versa) by exporting findings to a portable JSON.

14.4 Final word

SecAudit’s value is in the workflow: catalog → finding → risk severity → lifecycle → deliverable. The two builds are different packagings of that workflow, optimised for different engagements. The features documented in Part I are the contract between builds — both implement them, both honour the same brand language, both produce the same reports.
