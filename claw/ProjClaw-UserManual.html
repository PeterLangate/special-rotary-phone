Chapter 1 — Introduction

Chapter 1Meet ProjClaw

ProjClaw is a Microsoft Project add-in from kalmstrom.com Business Solutions. It opens a chat panel on the right side of your project, and from that panel you can ask an AI to build the work breakdown, assign resources, analyse the schedule, compare to baseline and generate reports — using plain English instead of the Project ribbon and custom-field dialogs.

1.1 What it is

ProjClaw is a VSTO add-in that runs inside the Microsoft Project desktop process. It drives your live schedule through the native Project object model — the same API Project's own UI uses — so every change is a real Project edit you can see, undo and save in the normal way. Every change runs through Project's scheduling engine: dependencies settle, the critical path updates, baselines compare correctly, earned value stays meaningful.

Under the hood it talks to Anthropic's Claude language model for conversation and reasoning. You supply your own Anthropic API key — the add-in does not run through any kalmstrom-hosted service.

1.2 What makes it different

Project as a schedule, not text

Other AI assistants treat project data as strings. ProjClaw treats it as tasks, resources, assignments and dependencies — all first-class objects.

Understands scheduling

Tasks, dependencies, resources, calendars, baselines, critical path, earned value — the domain vocabulary the agent needs for planning work.

Project's engine does the scheduling

ProjClaw never second-guesses dates. It asks Project to recalculate. The critical path, slack and earned value numbers you see are Project's own.

Seventy-plus specialist tools

Purpose-built for Project's idioms: constraints, deadlines, levelling, baselines, earned value, custom fields, calendars, views.

Chapter 2 — Installation & Setup

Chapter 2Installation & Setup

Installing ProjClaw takes about five minutes: install the add-in, get an Anthropic API key, and paste it into the settings window.

2.1 Install the add-in

  1. Close every open Project window.
  2. Run the ProjClaw installer (ProjClaw Setup.exe or the ClickOnce link supplied by kalmstrom.com).
  3. Follow the wizard — accept the prompts to install the Visual Studio Tools for Office (VSTO) runtime if it asks.
  4. Open Project. On the Task tab you should see a new ProjClaw button.
The button is missing? Go to File → Options → Add-ins. At the bottom, change Manage to COM Add-ins, click Go… and make sure ProjClaw is ticked. If it is in the Disabled Items list, re-enable it there.

2.2 Get your Anthropic API key

  1. Go to console.anthropic.com and sign in or create an account.
  2. Open the API Keys page.
  3. Click Create key, give it a name such as ProjClaw, and copy the key — it starts with sk-ant-.
  4. Add billing information. Anthropic charges pay-as-you-go; there is no monthly subscription required.
Cost in practice. Building a plan from scratch and running a critical-path analysis on a 200-task project typically costs a few cents per session. Very large plans (thousands of tasks) can use more tokens — scope by subtree or date range to keep it bounded.

Chapter 3Your first session

Open a blank project, click ProjClaw on the Task ribbon, and try a few of these. By the end of this chapter you'll have turned a narrative into tasks, linked dependencies, assigned a resource, and read the critical path.

  • Header — product name and a gear icon for settings.
  • Conversation — your messages on the right, ProjClaw's on the left.
  • Status bar — progress + queued tool calls + Cancel.
  • Input box — Ctrl+Enter or Send to submit.

3.2 A guided example

Try these in order.

Build a tiny plan

Add three tasks under a summary task called "Prototype": "Design" (3 days), "Build" (5 days), "Test" (2 days). Make them finish-to-start in that order.

Assign a resource

Add a resource called "Lead Engineer" at 100% and assign them to the Build task.

Find the critical path

Which tasks are on the critical path, and how much total slack does each of the others have?

Baseline compare

Save the baseline. Then push the Build task out by two days and tell me what happened to finish dates relative to baseline.

3.3 How ProjClaw thinks

Each instruction is sent to Claude along with a short description of the current project context (file name, open view, selected task). Claude breaks the request into one or more tool calls — add task, link dependency, assign resource, read field — which ProjClaw runs against the live project. After each batch, Claude sees the result and decides whether it's done.

3.4 Undo and safety

Every ProjClaw edit is a normal Project edit, so Ctrl+Z undoes it. ProjClaw groups many edits into one undo scope when you ask for something big ("build a WBS from this narrative") so you can roll back a whole plan with one keystroke.

Destructive ops confirm. Deleting a task, deleting a resource, or overwriting a baseline is always gated behind an explicit yes / no prompt.

Chapter 4 — Driving Project

Chapter 4Driving Project

This chapter walks through what ProjClaw can actually do, organised by task. Each section lists the capability, gives a few example prompts, and notes where Project's own idioms matter.

4.1 Tasks

Tasks are the fundamental unit of a Project plan. ProjClaw covers the full lifecycle.

  • Add. "Add a task 'Kick-off meeting', 2 hours, starting tomorrow morning." For bulk work, "turn these ten bullet points into tasks under the Prototype summary" calls bulk_add_tasks with the list.
  • Find, list, detail. "Find the task called 'Integration test'" / "Show task 14 in full — duration, percent complete, predecessors, successors, resources."
  • Rename and update. "Rename 'Kick-off meeting' to 'Project kick-off'." Any task field ProjClaw knows about can be set via set_task_field.
  • Delete. "Delete task 12." Destructive — ProjClaw confirms first.

4.2 Hierarchy

  • Indent and outdent. "Indent task 7 under task 6" / "Outdent these three tasks back to the top level."
  • Tree. "Show me the task hierarchy tree" — a nested outline of the whole plan.

4.3 Dependencies

Dependencies are how Project derives the schedule. ProjClaw manages them directly.

  • Link one pair. "Make Test finish-to-start after Build with a 1-day lag."
  • Chain a list. "Chain-link these five tasks in the order I listed them" — link_tasks_chain is faster than pairing them up one by one.
  • Remove. "Drop the dependency between Design and Build."
  • Predecessors and successors. "What depends on task 14?" / "What does task 14 depend on?"
  • Slack. "How much free slack does task 22 have?"
  • Critical path. "List the tasks on the critical path."

4.4 Task properties

  • Milestone. "Make the 'Go live' task a milestone."
  • Task type. "Set all design tasks to Fixed Work."
  • Notes. "Add a note to task 7: 'Requires sign-off from Legal.'"
  • Priority. "Set the priority of every 'blocker' task to 1000."
  • Constraint. "Set Must Start On 15 May for the Kick-off task."
  • Deadline. "Deadline on 'Go live' is 30 June."

4.5 Progress

Tracking progress is a daily activity in a running project; ProjClaw drives the four main knobs.

  • Percent complete. "Mark task 3 at 75% complete."
  • Actual start / finish. "Task 5 actually started on Monday" / "Task 7 finished yesterday."
  • Actual work. "Log 12 hours of actual work on task 9 for this week."

4.6 Analysis

  • Overdue. "Which incomplete tasks are past their finish date?"
  • In a date range. "List every task scheduled in July."
  • Cost per task. "What is the total cost of the Design phase tasks?"
  • Milestones. "Show me every milestone and its date."

4.7 Resources

  • CRUD. Add, delete, list, set any field — "Add 'Lead Engineer' as a work resource at 100%" / "Delete resource 'Placeholder'" / "Change the cost rate of 'Designer' to £80 / hour."
  • Max units. "Set max units on 'Designer' to 50% — they're half-time now."
  • Type. "Mark 'Materials' as a Cost resource, not Work."
  • Level resources. "Level the whole project using default settings." Project's own levelling engine resolves over-allocations.
  • Find over-allocated. "Which resources are over-allocated this month?"

4.8 Assignments

  • Assign. "Assign Designer to task 5 at 50% units."
  • Bulk assign. "Assign Lead Engineer to every task in the Prototype phase at 100%."
  • Remove. "Remove Designer from task 9."
  • Set units or work. "Set assignment work on task 14 to 20 hours."
  • List per task. "Who's assigned to task 22?"

4.9 Calendars

Project calendars define the working week. ProjClaw lists them, edits working time, and switches the project's active calendar.

  • List. "What calendars does this project have?"
  • Working time. "Set Fridays to half-days on the Standard calendar."
  • Non-working time. "Add a company holiday on 1 May to the project calendar."
  • Project calendar. "Switch the project to use the 'Night shift' calendar."

4.10 Baselines and variance

Baselines are first-class in Project, and ProjClaw surfaces the full reporting stack.

  • Save a baseline. "Save the current plan as Baseline 1."
  • Compare. "For every task whose finish has slipped vs. baseline, show by how many days."
  • Variance. "What is the schedule variance on task 5?"
  • Earned value. "What are BCWS, BCWP and ACWP for the project as of today?"

4.11 Costs

  • Per-task cost. "What does task 7 cost when I roll in resource rates and the fixed cost?"
  • Fixed cost. "Set a fixed cost of £2000 on task 3."
  • Project cost. "Total project cost, and cost-to-date."

4.12 Status reporting

  • Status date. "Set the status date to last Friday."
  • Status report. "Generate a status report for the Design phase: what's done, what's in progress, what's overdue."

Chapter 5 — Settings

Chapter 5Settings

ProjClaw's settings window opens from the gear icon in the chat panel header.

5.1 API key

Your Anthropic API key. Stored encrypted with Windows DPAPI under %APPDATA%\ProjClaw\. Click Show to reveal it temporarily.

Rotate your key if you share a machine. DPAPI protects keys from other Windows accounts on the same machine, but if you allow another person to use your Windows session, they can use your API key. Generate a new key in the Anthropic console and paste it here.

5.2 Model

Chooses which Claude model answers your messages.

ModelWhen to use
HaikuFast and cheap. Good for routine tasks with small inputs and short replies.
SonnetBalanced default. Handles summarisation, multi-step reasoning and longer drafts.
OpusHighest quality, slower and more expensive. Use for nuanced tone, sensitive material or hard judgement calls.

5.3 Confirmation policy

Controls how ProjClaw treats write operations.

  • Always ask (default) — every write operation requires confirmation.
  • Ask for destructive only — adds and edits run without a prompt; deleting a task / resource / baseline still confirms.
  • Never ask — not recommended for shared or production plans.
Delete and baseline overwrite always confirm. Regardless of this setting, destructive operations are gated behind an explicit yes / no prompt. This is hard-coded and cannot be disabled.

5.4 Logging

Controls the local log file used for diagnostics.

  • Off — no log written.
  • Errors only (default) — only failures are recorded.
  • Verbose — every tool call, input and result is written to the log.

Log file location: %APPDATA%\ProjClaw\projclaw.log. It is a plain text file you can open in Notepad. Document, mailbox or workbook contents are not logged at any verbosity level.

5.5 Panel position

By default the chat panel docks on the right of the main Project window. You can float it or dock it on the left. The position is remembered per Windows profile.

Chapter 6 — Tips & troubleshooting

Chapter 6Tips & troubleshooting

6.1 Prompt patterns that work

  • Scope your operations. "In the Prototype phase…" or "for tasks 10–25…" narrows edits and keeps reads fast on large plans.
  • Ask for reads before writes. List the tasks you're about to change; confirm the names are right; then issue the write.
  • Wrap big changes in an undo scope. "Begin an undo scope, turn this statement of work into a WBS, then end the scope" gives you a single Ctrl+Z to roll back the whole plan.
  • Mind the status date. Earned-value and status-report tools answer relative to Project's status date — set it explicitly for a report.

6.2 Common pitfalls

SymptomCauseFix
Dates don't shift after a dependency changeCalculation mode is manual.Switch to automatic via File → Options → Schedule, or ask ProjClaw to recalculate.
Critical path looks wrongProject's "days to critical" threshold is non-zero.In Project's options, set "Tasks are critical if slack is ≤ 0 days".
Resource over-allocation not detectedLevelling or per-resource calendar overrides are in play.Ask ProjClaw to list the resource's working times and its current assignments side by side.
Custom field values look oddThe field type (Text1…Text30, Number1…, Cost1…) doesn't match what you're writing.Ask ProjClaw for the field's type before writing.

6.3 Cost and token tips

  • Listing tasks is cheap. Bulk-adding hundreds of tasks runs multiple tool calls — batch by phase to keep the conversation tight.
  • Haiku handles routine add / link / read work. Sonnet when you want reasoning about structure ("is this plan front-loaded or back-loaded?"). Opus only when tone or nuance really matters.

6.4 Bug-report helpers

Two built-in tools make it easy to gather the build fingerprint and capability surface when you need to report an issue:

  • get_version_info — ask "what version of ProjClaw am I running?" and ProjClaw replies with add-in name, auto-refreshed build version, the Project host version, OS, and .NET runtime. The same version number is always visible in the top-right corner of the Settings dialog, so you can grab it without sending a chat turn.
  • list_tools — ask "which ProjClaw tools mention 'baseline'?" and it filters the full tool catalogue by any substring of the name or description. Useful when you're not sure whether a capability exists, or when you want to compare what's available against the online docs.

Chapter 7 — Advanced operations

Chapter 7Advanced operations

Chapter 4 covers the day-to-day surface: tasks, resources, assignments, baselines. This chapter covers the operations people reach for less often — cross-project consolidation, template authoring, calendar creation, scheduling mode, and built-in reports. Every operation here still goes through a normal chat prompt; ProjClaw picks the right tool from the wording.

7.1 Multi-project consolidation

Project supports a master plan that inserts one or more sub-projects as rows. ProjClaw drives this from chat.

  • Insert a sub-project into the current master. "Insert C:\Plans\Phase1.mpp as a sub-project before the 'Integration' summary task." ProjClaw opens the file, links it into the active master, and preserves the link so both files update together.
  • Consolidate several files into a brand-new master. "Create a new master plan and pull in these four .mpp files: Phase1.mpp, Phase2.mpp, Phase3.mpp, Phase4.mpp." ProjClaw creates an empty master and appends each path as an inserted sub-project.
  • Read-only inserts. Add "as read-only" to the prompt when you want the sub-project to display without allowing edits to flow back to the source file.

Tip. Sub-project paths must be absolute. Relative paths like ..\Phase1.mpp are rejected by Project's COM layer; ProjClaw surfaces that as an error rather than silently normalising.

7.2 Saving a plan as a reusable template

A .mpt template is the shape of a plan — tasks, resources, calendars, views — without the values you've accumulated. Useful when you want to spin up next quarter's plan from last quarter's structure.

  • Plain save. "Save this as a template at C:\Templates\Quarterly.mpt."
  • Scrub on the way out. "Save a template of this plan, but strip the baselines, actuals, resource rates, and fixed costs." Project applies each scrubber during the save so the template ships clean.

Environment caveat. Older Project builds reject the 14-argument FileSaveAs signature that carries the scrub flags. If that happens, ProjClaw falls back to a plain 2-argument save — the .mpt is still written, but the scrub flags aren't applied. The tool result string will say "(fallback: 2-arg FileSaveAs; scrub flags not applied)" so you know to clear the values manually.

7.3 Creating and editing calendars

Chapter 4 covers editing working time on an existing calendar. ProjClaw also creates calendars from scratch.

  • Create blank. "Create a new base calendar called 'Night shift'." The calendar starts with Project's default working week; edit working time afterwards.
  • Create by copy. "Create a 'Client holidays' calendar based on the Standard calendar." The copy inherits the source calendar's working time and exceptions; you then add client-specific non-working days.
  • Set working time. "On 'Night shift', set Monday–Friday shift 1 to 22:00–06:00." Multi-shift prompts work too: "shift 2 from 14:00 to 22:00."
  • Non-working exceptions. "Add a non-working exception 'Client blackout' on 1–7 August to the 'Client holidays' calendar."
  • Assign to the project. "Make 'Night shift' the project calendar" / "Set resource 'Alice' to use the 'Night shift' calendar."

7.4 Scheduling mode

Project lets you schedule forward from a start date or backward from a finish date. ProjClaw toggles this directly.

  • Schedule from start (default). "Schedule this project from its start date, 1 June." Tasks flow forward; the finish date is computed.
  • Schedule from finish. "Schedule backward from the 'Go live' date, 30 September." Tasks flow backward; the start date is computed. Useful for fixed-deadline plans.
  • Project start / finish date. Independent of mode: "Set the project start date to 1 July" or "set the project finish to 31 December."

7.5 Manual vs auto recalculation

  • Switch to manual calc. "Turn calculation off" — useful when you're about to make a large batch of edits and don't want Project recomputing after every one.
  • Switch back to automatic. "Turn calculation back on."
  • Force a recalc now. "Recalculate the project." No-op if auto-calc is on.

7.6 WBS renumbering

WBS codes rarely survive heavy editing intact. When they drift, ProjClaw re-applies the WBS mask across every task.

  • Renumber. "Renumber the WBS codes across the whole plan." Uses Project's own WBSCodeRenumber, so the result matches what the UI's Renumber command would produce.

7.7 Built-in reports

Project ships a catalogue of canned reports (Burndown, Cashflow, Cost overview, Resource overview, etc.). ProjClaw drives them by name.

  • List available reports. "What reports does this plan offer?"
  • Open a report. "Show the 'Cashflow' report." Project opens the report in the active view; close it via Project's UI when done.
  • Export a report to PDF. "Export the 'Cost overview' report as PDF to C:\Reports\cost.pdf." ProjClaw routes via DocumentExport, not FileSaveAs, because PDF isn't a PjFileFormat on modern Project builds.
