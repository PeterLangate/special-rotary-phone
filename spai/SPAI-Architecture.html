Chapter 1Overview

SPAI is an SPFx Application Customizer that injects a floating AI assistant into every SharePoint page — Online or on-premise (SP 2019). A floating avatar opens a slide-in side panel where users converse with an AI provider of their choice, which reads, analyses, and writes SharePoint data using natural language.

1.1 Dual-Build Architecture

SPAI mirrors the TemplatesManager project structure: a thin SPFx shell injects a separately-built webpack bundle. This decouples the SharePoint deployment from the core logic, enabling future SP on-premise deployments without re-packaging.

SPAISPFX\ — SPFx Shell

SPFx 1.22.2 Application Customizer (heft). Establishes page context, then calls into SPAI.js — bundled directly via webpack import.

SPAISolution\ — Core Bundle

Webpack-built TypeScript. All UI, Claude API, and SharePoint REST logic lives here. All classes exposed on window.KDebug.SPAI.

1.3 Technology Stack

SPFx 1.22.2

Application Customizer. Node ≥ 22.14, TypeScript ~5.8, heft build. No React. Appears on every SharePoint page — Online and on-premise.

Admin Shared Library

X:\Source\Admin — kSite, kSPRestCore, kEditableTable, kHTMLSideBar, kAvatar, kModalDialog. All tested against SP 2019.

Multi-Provider AI

Supports Claude (Anthropic), OpenAI (GPT-4o etc.), Azure OpenAI, Google Gemini, and Ollama (local). All providers implement IAiProvider. The active provider is selected in a one-time setup form and stored in AI Findings/config.json.

Mermaid.js

In-browser diagram rendering for flow charts and relationships. Initialized once per session with Office Fluent theme colours.

1.4 SharePoint Compatibility

All SharePoint REST calls use the standard /_api/ endpoint — tested against SP 2019. Microsoft Graph API is not used (unavailable on-premise). Search uses the classic SharePoint Search API (/_api/search/query), which is available on all supported versions.

Namespace: All SPAI classes are exposed on window.KDebug.SPAI — the same root namespace as TemplatesManager — keeping the shared code management consistent across projects.

Chapter 2Architecture

The core SPAI experience runs entirely in the browser — no Azure Functions, no proxy server. The AI provider API is called directly from SharePoint, tool results are computed locally, and the configuration is stored in a provisioned SharePoint document library on each site. An optional Microsoft 365 Copilot agent (section 2.4) provides the same tools through a Teams bot with interactive Adaptive Cards.

2.1 Direct API — No Backend

Every message exchange flows from the browser to the configured AI provider and back. Tool calls (reading lists, searching, writing items) execute in the same browser session using the existing SharePoint cookie auth — no SharePoint credentials are forwarded to the AI provider.

Browser (SPFx)
  → AI provider API (Claude / OpenAI / Azure / Gemini / Ollama)
      Header: x-api-key: <from AI Findings/config.json>
  ← SSE stream or JSON response
  → execute tool calls via SharePoint REST /_api/
  → next turn with tool results
  ← … repeat until stop_reason === "end_turn"

2.2 The "AI Findings" Library

On first use, SPAI provisions a document library named AI Findings in the current site. The library serves three purposes:

Configuration

config.json in the root. Stores the selected AI provider, its API key or endpoint, optional model override, and custom prompt. Readable by site members, writable by owners.

Site Instructions

instructions.md files give the AI site-specific context. Loaded at startup and injected into the system prompt.

Cache

Analysis results (site structure, user rosters) stored as JSON to avoid redundant REST calls across sessions.

Provisioning: ApiKeyLoader calls kSite.CreateMissingReqListsAndFields on first save. If config.json is missing or contains no key for the selected provider, a setup form in the side panel collects the configuration and saves it via REST Files/Add.

2.3 Provider Abstraction Layer

All AI providers implement the IAiProvider interface. ConversationManager is fully provider-agnostic — it sends an AiRequest and receives a stream of canonical AiEvent objects regardless of which provider is active.

Canonical AiEvent stream

Each provider translates its wire events into the same five canonical event kinds. ConversationManager routes these regardless of which provider fired them:

Event kindAction in ConversationManager
text_deltaAppend text to current assistant bubble (word-by-word streaming)
tool_startFlush accumulated text; begin capturing tool input
tool_deltaAppend chunk to tool input buffer
tool_endParse tool input JSON; push completed AiToolCall to content
stop { reason: "tool_use" }Execute all tool calls; send results in next turn (loop)
stop { reason: "end_turn" }Conversation turn complete; fire OnTurnDone callback

2.4 Microsoft 365 Copilot Agent (SPAICopilot)

In addition to the browser-based SPFx extension, SPAI is available as a custom engine agent inside Microsoft 365 Copilot. This is a separate deployment — a Teams bot that provides the same SharePoint tools through interactive Adaptive Cards.

Why a Teams bot instead of an API plugin?

We evaluated two architectures for the Copilot integration:

CapabilityAPI Plugin (rejected)Teams Bot (chosen)
Adaptive CardsDisplay-only citationsFull interactivity — forms, buttons, updates
Write confirmationBasic text promptRich card with details + Confirm/Cancel
User inputNatural language onlyForm fields — dropdowns, dates, text
Card updatesNot possibleDynamic replacement after action

API plugin Adaptive Cards cannot contain Action.Submit, Action.Execute, or Input elements — they are strictly display-only. Since SPAI has 27+ write operations needing rich confirmation and input forms, the bot architecture was required.

Architecture

The bot runs on Azure App Service and uses the Teams SDK v2. It reuses the same SharePoint REST endpoints as the browser-side SPAI, authenticated via OAuth 2.0 on-behalf-of (OBO) token exchange from the user's Teams SSO token.

User in Teams / M365 Copilot
  → Teams Bot Service → SPAI Bot (Azure App Service)
    → SSO token → OBO → SharePoint REST /_api/
    → Adaptive Card response (fully interactive)
Independent deployment: The Copilot agent and the SPFx extension are completely independent. The SPFx extension continues to work in SharePoint without the bot, and the bot works in Teams/Copilot without the SPFx extension.

3.2 Write Confirmation Pattern

Before any write tool (create_list_item, update_list_item, delete_list_item) calls the Admin library, toolExecutor.ts inserts a ConfirmAction bubble into the message thread. The REST call is suspended until the user clicks Accept or Cancel.

  1. Claude calls a write tool with proposed field values
  2. toolExecutor.ts renders a ConfirmAction bubble showing the operation details
  3. User clicks Accept — the Admin REST call fires; result returned to Claude
  4. User clicks Cancel — a synthetic error result is returned; Claude acknowledges gracefully
Important: Claude is instructed via the system prompt to always explain what it is about to do before calling a write tool. The confirmation bubble is a safety net — not a substitute for clear communication.

3.3 Microsoft 365 Copilot Search

When enabled, SPAI can search across the entire Microsoft 365 tenant using Microsoft's semantic index — the same engine that powers Copilot. This goes beyond the current site's classic search.

ToolScopeNotes
search_m365_contentAll SharePoint sites + OneDriveCross-tenant semantic search with KQL filter support
search_my_onedriveCurrent user's OneDriveSearches the user's personal files only

Both tools call the Microsoft Graph Copilot Retrieval API (POST /v1.0/copilot/retrieval) and return text extracts with relevance scores, titles, authors, and modification dates.

Prerequisites:
  • Each user must have a Microsoft 365 Copilot license
  • A SharePoint admin must approve Graph API permissions: Files.Read.All, Sites.Read.All
  • The site administrator must enable "Enable M365 Copilot search" in SPAI settings

If a user without a Copilot license triggers the tool, SPAI returns a clear error message and automatically disables M365 search for the rest of the session to prevent repeated failures.

3.4 Future Plans

The following capabilities are planned for future releases. They require additional platform permissions or infrastructure that is not yet in place.

Communication on behalf of the user

SPAI currently has no ability to send messages or notifications. Future releases aim to let Claude act as a communication proxy for the signed-in user, including:

  • Send email — compose and send emails via Microsoft Graph (POST /me/sendMail) on behalf of the signed-in user. Example: "Send a summary of overdue tasks to Marcus."
  • Send Teams messages — post a chat message or channel message via Graph on behalf of the user. Example: "Notify the project team that the deadline has moved."
  • Create calendar events — schedule follow-up meetings or reminders via Graph. Example: "Book a review meeting with the task owners next week."
Prerequisite: All communication tools require delegated Graph API permissions (Mail.Send, Chat.ReadWrite, Calendars.ReadWrite) granted to the SPAI application registration. The user must explicitly consent before any message is sent on their behalf.

Proactive notifications

Allow SPAI to surface alerts automatically — e.g. items overdue by more than N days, or a list field matching a threshold — without the user asking first.

File operations

Read and summarise document library files (PDF, Word, Excel) via Graph and SharePoint REST, so Claude can answer questions about document content in addition to list data.

4.5 kEditableTable

kEditableTable uses a light theme by default — no CSS overrides needed. render_grid in toolExecutor.ts creates the table, sets columns (type kHTMLEditableColType), binds rows, and calls ToTable() to render.

Memory leak guard: kEditableTable starts a 200 ms setInterval on construction. Always call Grid.Destroy() when the panel closes or the message containing the grid is removed from the DOM.

Chapter 5Setup & Development

SPAI uses a dual-build approach. The SPFx shell (SPAISPFX\) is a thin heft project that loads context and calls into the core bundle. The core bundle (SPAISolution\) is built with webpack and bundled directly into the SPFx package via a static import — the same pattern used by TemplatesManager.

5.1 SPFx Shell — Entry Point

// SPAISPFX/src/SpaiAssistant.ts
import { BaseApplicationCustomizer, PlaceholderName }
    from "@microsoft/sp-application-base";
import { kSPFXAppContext }
    from "../../Admin/ReUsableTS/kSPFXAppCustomizer";

export default class SpaiAssistant extends BaseApplicationCustomizer {

    public onInit = (): Promise<void> => {
        kSPFXAppContext.FromSFPXContext(this);
        this._injectCoreBundle();
        kSPFXAppContext.OnContextChange =
            () => window.KDebug?.SPAI?.OnNavigate();
        return Promise.resolve();
    }

    private _injectCoreBundle = (): void => {
        const bottom = this.context.placeholderProvider
            .tryCreateContent(PlaceholderName.Bottom);
        if (!bottom) return;
        const script = document.createElement("script");
        script.src = this._resolveBundleUrl();
        script.onload = () => window.KDebug?.SPAI?.Init(bottom.domElement);
        document.head.appendChild(script);
    }

    private _resolveBundleUrl = (): string => {
        const baseUrl =
            this.context.pageContext.legacyPageContext.cdnPrefix as string;
        return `${baseUrl}SPAI.js`;
    }

    public onDispose = (): void => {
        kSPFXAppContext.StopURLMonitor();
        window.KDebug?.SPAI?.Destroy();
    }
}

5.4 Copilot Bot Deployment (one-click)

The SPAI Copilot bot is deployed via a single PowerShell script that automates all Azure and Entra ID setup:

cd X:\Source\SPAI\SPAICopilot
.\Deploy-SpaiCopilotBot.ps1 -TenantName "contoso"

The script performs 5 phases automatically:

  1. Prerequisites — checks for Azure CLI and Node.js; installs via winget if missing
  2. Azure login — verifies az login session and tenant
  3. Entra ID — creates app registration, client secret, API permissions (Graph + SharePoint delegated), admin consent, Application ID URI for Teams SSO
  4. Azure infrastructure — resource group, App Service plan (B1), web app (Node 22), Azure Bot Service, Teams channel
  5. Build & deploy — npm install, TypeScript build, zip deploy to App Service

The script outputs a Teams app package (manifests/spai-copilot-bot.zip) ready for sideloading in Teams.

Manual step required: After deployment, add the two Teams authorized client apps in the Entra portal (Expose an API section). This cannot be automated via Azure CLI. The script prints the exact GUIDs to add.

Cost: Azure App Service B1 (~$13/month) + Azure Bot Service Free tier ($0). Total: ~$13/month for testing, ~$73/month for production (S1).

Chapter 6AI Provider Configuration

SPAI supports five AI providers. The provider is selected once in the setup form, and its credentials are stored in AI Findings/config.json. Switching provider is as simple as opening the setup form again and saving a new selection.

6.1 Supported Providers

Claude (Anthropic)

Default provider. SSE streaming. Requires a sk-ant-… API key from console.anthropic.com. Default model: claude-opus-4-6.

OpenAI (GPT-4o etc.)

SSE streaming. Requires a sk-… API key from platform.openai.com. Default model: gpt-4o. Supports any chat model via the model override field.

Azure OpenAI

SSE streaming via a custom endpoint. Requires a full Azure endpoint URL (including deployment name and api-version query parameter) and a resource API key from the Azure portal.

Google Gemini

REST JSON (non-streaming). Requires an AIza… API key from aistudio.google.com. Default model: gemini-2.0-flash.

Ollama (Local)

Runs models locally — no API key required. Default endpoint: http://localhost:11434. The model must be pulled with ollama pull <model> before use. Set the model name in the model override field.

6.2 First-Time Setup

On first click of the SPAI avatar, if no valid config.json exists in the AI Findings library, the setup form opens automatically inside the side panel:

  1. Select a provider from the dropdown — only the relevant fields appear
  2. Enter the API key (or endpoint URL for Azure / Ollama)
  3. Optionally enter a Model ID to override the provider default
  4. Optionally enter a Custom Prompt — runs silently the first time the panel opens each session (e.g. "Call get_site_overview and render_grid the results.")
  5. Click Save & Open — settings are saved to AI Findings/config.json and the chat panel opens immediately
Re-opening setup: The setup form only shows automatically when no valid config exists. To change the provider or key after initial setup, delete or edit config.json in the AI Findings document library.

6.3 Config File Format

The configuration is stored as plain JSON in AI Findings/config.json on the SharePoint site. All fields are strings. Unused provider fields are saved as empty strings.

{
  "ai_provider":     "claude",
  "claude_api_key":  "sk-ant-...",
  "openai_api_key":  "",
  "azure_endpoint":  "",
  "azure_api_key":   "",
  "gemini_api_key":  "",
  "ollama_endpoint": "http://localhost:11434",
  "model_override":  "",
  "custom_prompt":   ""
}
Backwards compatibility: Existing config.json files that only contain anthropic_api_key (the previous format) are automatically read as Claude provider with no migration needed.

6.5 Custom Prompt

The custom prompt runs silently as the first user message when the chat panel opens — before any human input. Use it to automate a greeting sequence or an initial data load:

Once per session: The custom prompt only runs if the conversation thread is empty. Opening and closing the panel within the same session does not re-run it.
