4. Azure Functions

FunctionTriggerBeskrivningFas

§ 11 — Deployment architecture

NEXT består av tre lager: SharePoint Online (hub-site + ett projekt-subsite per projekt), ett paket med SPFx-webbdelar (ProjectHub, Dashboard, AO, ÄTA, Budget m.fl.) samt ett set Azure Functions som utför provisionering, OCR och Fortnox-integration. All persistens sker i SharePoint — Azure Functions är statslösa.

11.1 Site-topologi

NEXT Hub Site (/sites/next) ├── Projektkatalog (hub-template.xml) ├── Leverantörer (hub-template.xml) └── Sub-sites per project (created on demand by ProjectProvisioner AF) └── /sites/next/NEXT-2024-001 ├── Arbetsorder ├── ÄTA-hantering ├── Avvikelser ├── Dagbok ├── Tidsrapportering ├── Budget ├── Kundfakturor ├── Leverantörsfakturor └── Ritningar (document library)

11.2 Azure Functions

FunctionTriggerAnsvar
NEXT-HubBootstrap Manual (one-shot) Applicerar hub-template.xml på hub-siten. Körs en gång vid initial installation.
NEXT-ProjectProvisioner HTTP POST Skapar sub-site, applicerar pnp-template.xml, skapar SP-grupper (Owners/PL/Ekonomi/Fält/Läsare), lägger in rad i Projektkatalog. Se src/functions/ProjectProvisioner/index.ts.
NEXT-InvoiceOcrParser Blob created (SharePoint drop-folder) Läser PDF-faktura, kör Azure Form Recognizer, skapar rad i Leverantörsfakturor med OCRConfidence-score. Låg confidence flaggar raden för manuell granskning.
NEXT-FortnoxSync Timer (var 30:e min) + HTTP Tvåvägssynk mot Fortnox. Skickar nya kundfakturor, hämtar betalstatus, uppdaterar FortnoxID och InvoiceStatus. HTTP-triggern används av "Synka Fortnox"-knappen i UI.
NEXT-DwgConverter Blob created (Ritningar library) Konverterar uppladdade .dwg/.pdf-ritningar till SVG, skriver SVG-url till SVGUrl-fältet för inline-visning i Ritning-vyn.
NEXT-NotificationDispatcher Timer (daglig) + SP webhook Skickar Teams/e-post när: ÄTA väntar godkännande > 3 dagar, avvikelse med Severity=Kritisk skapas, kundfaktura blir försenad, OCR-confidence < 75%.

11.3 Dataflöde — nytt projekt

11.4 Identitet och hemligheter

Azure Functions körs med system-assigned Managed Identity. MI har Sites.FullControl.All via Graph och Sites.Manage.All via SharePoint REST enbart på NEXT-hub-siten och dess sub-sites (scoped via SP App Catalog). Fortnox-token lagras i Azure Key Vault och läses via referens i local.settings.json/App Settings (@Microsoft.KeyVault(...)). Inga hemligheter i källkod.

11.5 CI/CD

Azure DevOps pipelines: NEXT-SPFx-build (gulp bundle + ship, publicerar .sppkg till App Catalog), NEXT-AzureFunctions-deploy (tsc build + zip-deploy till Function App). Provisioneringstemplates (hub-template.xml, pnp-template.xml) deployas som artefakter och läses av respektive AF via Blob Storage.

§ 12 — Test strategy

NEXT har fyra typer av test med tydliga ansvarsområden. Ingen kodändring får mergas utan minst en grön körning på respektive nivå. CI-pipelinen NEXT-CI kör enhets- och integrationstest på varje PR; Playwright-testen körs nattligen mot en dedikerad test-hub https://langate.sharepoint.com/sites/next-test.

12.1 Testpyramid

NivåVerktygScopeTäcker
Unit Jest (SPFx)
vitest (AF)		 1 klass / 1 funktion Rena beräkningar, OCR-confidence-klassificering, budget-aggregering, Fortnox-radmappning, kErrorMode-guards i alla metoder. Mål: > 80% branch coverage i src/.
Integration vitest + @azure/functions-test En AF mot SP-stub ProjectProvisioner hela flöden: parse → Graph-stub → PnP-stub → hub-insert → respond. Verifierar att felinjektion på valfritt steg triggar korrekt Fail()-respons.
Provisioning Pester (PowerShell) PnP-template.xml + hub-template.xml Apply → verify alla 9+2 listor finns, fältnamn matchar types.ts, Choice-värden är korrekta, View-queryn returnerar förväntade fält. Körs i en egen wipe-site vid varje körning.
End-to-end Playwright Hela SPFx-web­appen Golden-path-flöden: skapa projekt (wizard → progress modal → ny rad i Projektkatalog), OCR-tolka faktura, godkänn ÄTA, Fortnox-synk. Kör nattligen. Regressionsmat­ris för IE/Edge/Chrome är medvetet inte inkluderad — NEXT levereras endast för Edge/Chrome.

12.2 Fixtur­data

Fixtur­skriptet scripts/seed-test-hub.ps1 skapar ett referens-projekt NEXT-TEST-001 "Fixturprojekt" med 4 arbetsordrar, 2 ÄTA, 3 avvikelser, 2 veckors tidrapport, 8 budgetposter och 4 leverantörsfakturor. Samma fixturset används av både Pester och Playwright — en enda sanningskälla för testdata.

12.3 CI-pipeline

PR opened ───► NEXT-CI (Azure DevOps) │ ├─► tsc --noEmit (type check) ├─► eslint src/**/*.ts (lint — no anonymous functions, no ternaries, etc.) ├─► jest --ci (SPFx unit) ├─► vitest run (AF unit + integration) └─► gate: all green → allow merge to main nightly ───► NEXT-E2E (nightly at 02:00) │ ├─► seed-test-hub.ps1 (reset + reseed fixture project) ├─► Invoke-Pester (provisioning check) ├─► playwright test (golden-path e2e) └─► Teams notification if any step fails

12.4 Uppdaterings­krav

Ingen PR mergas utan: (1) motsvarande enhetstest, (2) grön CI-körning, (3) uppdatering av denna spec om PR ändrar list­schema, permissions, eller Azure Function-kontrakt. Dokumentation och tester är inte valfria — de är del av leveransen.

§ 13 — Data migration plan

Langate har idag projekt­data spridd över Excel-filer på en fil­server (\\langate-fs01\projekt\*), delvis i Fortnox (fakturor), och delvis i fysiska pärmar för äldre projekt. NEXT ersätter Excel-delen helt och blir master för all icke-finansiell data framåt.

13.1 Scope

KällaDataStrategi
\\langate-fs01\projekt\*.xlsx Aktiva projekt (~15 st): arbetsordrar, budget, tidrapporter, dagbok Aktiv migrering via mappnings­mall + staging-lista (se 13.3)
\\langate-fs01\ritningar\*.dwg DWG-ritningar per projekt Kopiera till projekt-sub-sitets Ritningar-bibliotek, konvertering till SVG sker automatiskt av DwgConverter AF
Fortnox Kund- och leverantörsfakturor Ingen migrering — Fortnox är redan master (ADR 0003). NEXT cachar efter cut-over.
Pärmar (arkiv) Avslutade projekt före 2024 Ingen migrering — historiska projekt stannar i pärmform. Referens­lista i Projektkatalog med Status=Avslutad och tom sub-site-URL.

13.2 Mappnings­mall

Varje projekt-PL får mallen NEXT-Migration-Template.xlsx med 9 flikar (en per lista). Första raden innehåller fält­namn som matchar types.ts exakt (ProjectID, AONumber, Priority, StartDate, osv.) Andra raden är fält­beskrivning. Rad 3 och nedåt är data.

PL får 2 veckor att fylla i sitt projekts data i mallen. Stöd ges av en "migration office hour" 2 ggr/vecka under samma period. Excel-mallen har inbyggda valideringsregler (Choice-fält som dropdowns, datum­format, obligatoriska kolumner) som speglar pnp-template.xml.

13.3 Staging-list-approach

PL fills NEXT-Migration-Template.xlsx │ ▼ NEXT-HubBootstrap site has a temporary list "Migration-Staging" per project │ (import via Invoke-PnPListItemImport from Excel) ▼ NEXT-MigrationValidator AF runs rule checks: ├─► Field-level: required fields filled, choice values valid, dates parseable ├─► Cross-row: ProjectID consistent, AO-numbers unique, budget rows sum correctly └─► Cross-list: ÄTA references existing AO, Tidsrapportering references existing AO │ ▼ Validator output: /sites/next/migration-reports/PROJECTID-validation.html ├─► GREEN rows: ready to commit └─► RED rows: PL must fix in the Excel and re-import │ ▼ PL reviews & approves │ ▼ NEXT-MigrationCommit AF reads Migration-Staging and writes rows to the real per-project sub-site lists (Arbetsorder, ÄTA, ...). Staging list is deleted on success.

13.5 Rollback-plan

Om cut-over misslyckas (validator-fel, performance­problem, data­korruption): NEXT-sub-sites bevaras men SPFx-appen döljs från landnings­sidan; PL fortsätter i Excel­mallarna. Root-cause-analys innan ny cut-over planeras. SharePoint-data tas inte bort — den ligger kvar som fallback för manuell åter­migrering.
