Kapitel 1 — Bakgrund och syfte

Kapitel 1Bakgrund och syfte

FortBridge är en integrationslösning som låter säljorganisationer arbeta i Dynamics 365 Sales utan att behöva hantera dubbel datainmatning i Fortnox. Systemet håller kunder, produkter, offerter och ordrar kontinuerligt i synk mellan de två plattformarna — med fördröjningar på sekunder till minuter.

1.1 Problemet som löses

Många svenska företag använder Fortnox som sitt ekonomisystem och Dynamics 365 Sales som sitt CRM-system. Utan integration uppstår ett klassiskt problem: säljteamet skapar en order i D365 Sales och ekonomiavdelningen måste sedan lägga in samma order manuellt i Fortnox. Resultatet:

  • Dubbelarbete som tar tid och skapar frustration hos båda avdelningarna
  • Fel och avvikelser när data registreras på olika sätt i de två systemen
  • Försenade fakturor eftersom ekonomiavdelningen är beroende av manuell kommunikation från sälj
  • Bristande synlighet — säljaren ser inte betalningsstatus direkt i CRM

Sammantaget innebär detta längre ledtider i faktureringen, ökad risk för kassaflödesproblem och ett sämre kundintryck — problem som FortBridge löser utan att kräva förändrade arbetsrutiner i säljteamet.

1.2 Vad är FortBridge?

FortBridge är en Azure-baserad integrationstjänst som körs i bakgrunden och håller de två systemen i synk. Lösningen är byggd som en Azure Function App (B1-plan) och kommunicerar med båda systemens REST API:er.

Fortnox

Svenskt molnbaserat ekonomisystem med ett väldokumenterat REST API v3. Hanterar kunder, fakturor, ordrar, artiklar, betalningar och bokföring.

Dynamics 365 Sales

Microsofts CRM-plattform för säljprocesser. Hanterar konton, kontakter, affärsmöjligheter, offerter och ordrar via Dataverse Web API.

Kapitel 2 — Fortnox API

Kapitel 2Fortnox API

Fortnox erbjuder ett väldokumenterat REST API (version 3) som ger bred tillgång till ekonomisystemets resurser. API:et stöder JSON och XML, och används av hundratals tredjepartsintegrationer på den svenska marknaden.

2.1 Teknisk grund

Bas-URL

https://api.fortnox.se/3/

Alla resurser nås via denna bas med resursnamn som sökväg, t.ex. /3/customers.

Autentisering

OAuth 2.0 Authorization Code Flow. Access Token gäller 1 timme, Refresh Token 60 dagar. Refresh Token roteras vid varje förnyelse.

Hastighetsgräns

400 anrop per minut per integration. Vid överskridning returneras HTTP 429 med Retry-After-header. FortBridge respekterar denna header.

Paginering

Max 500 poster per sida via limit och page. Svarsobjektet innehåller MetaInformation med @TotalPages och @TotalResources.

Effektiv polling: Nästan alla list-resurser stöder filtret ?lastmodified=YYYY-MM-DD. FortBridge sparar tidsstämpeln för senaste lyckade poll i Table Storage och skickar alltid detta filter — aldrig en full lista. Det håller anropsvolymen långt under hastighetsgränsen.

2.2 OAuth 2.0-flöde

Fortnox implementerar Authorization Code Flow. Installationsflödet körs en gång; därefter sköts tokenförnyelse automatiskt av FortBridge.

Kapitel 3 — Dynamics 365 Sales API

Kapitel 3Dynamics 365 Sales API

Dynamics 365 Sales exponerar all sin data via Dataverse Web API — ett OData v4-baserat REST API. FortBridge använder detta API för att läsa och skriva konton, produkter, offerter och ordrar. Utgående händelser skickas via Dataverse Webhooks (Service Endpoints).

3.1 Teknisk grund

Bas-URL

https://<org>.api.crm4.dynamics.com/api/data/v9.2/

Suffixet crm4 gäller för europeiska tenants. Korrekt URL hämtas alltid från Power Platform Admin Center under miljöinställningarna (fält: "Web API URL").

Autentisering S2S

Azure AD Application permissions (inte Delegated). Token begärs med scope https://<org>.crm4.dynamics.com/.default. Kräver en Application User i Dataverse.

OData-frågor

Stödjer $filter, $select, $expand och $orderby. Batch-anrop via /$batch rekommenderas vid synkronisering av orderrader.

Utgående händelser

Dataverse Webhooks (Service Endpoints av typen Webhook) registreras i Plugin Registration Tool och skickar HTTP POST till FortBridge vid Create/Update/StateChange.

Server-till-server-autentisering: För en tjänst utan inloggad användare ska alltid Application permissions användas — inte Delegated permissions (user_impersonation). Skapa en Application User i Dataverse, koppla den till appregistreringen via Client ID, och tilldela en säkerhetsroll med CRUD-rättigheter på relevanta entiteter.

3.2 Dataverse Webhooks — registrering

D365 skickar händelser till FortBridge via Webhooks registrerade i Plugin Registration Tool. Processen:

  1. Öppna Plugin Registration Tool → Register New Service Endpoint → välj typ Webhook.
  2. Fyll i FortBridges endpoint-URL och konfigurera autentisering som HttpHeader med headern X-FortBridge-Secret.
  3. Lägg till ett Step på Service Endpoint (ej på ett Plugin): välj entitet, message och stage. Använd PostOperation, Asynchronous för bättre prestanda och tillförlitlighet.
  4. D365 förväntar sig HTTP 200-svar inom 60 sekunder — annars anses leveransen misslyckad och steget kan automatiskt inaktiveras vid upprepade fel.

Kapitel 4 — Arkitektur

Kapitel 4Arkitektur

FortBridge körs som en Azure Function App på B1-plan. Lösningen innehåller åtta funktioner (triggers) som hanterar synkronisering i båda riktningarna, tokenhantering och felåterförsök. All state lagras i Azure Table Storage och alla hemligheter i Azure Key Vault.

4.2 Azure Function App — B1-plan

B1 Basic — specifikation

1 kärna, 1,75 GB RAM, 10 GB diskutrymme. Stöder Always On (förhindrar viloläge) — kritiskt för timer-triggers. Ca 530 kr/månad i West Europe.

Varför B1 och inte Consumption?

Consumption-plan garanterar inte exakta timer-intervall och går i viloläge vid inaktivitet. B1 ger förutsägbar latens och exakta 5-minuterspollar.

4.4 Statlagring med Azure Table Storage

FortBridge använder Azure Table Storage för att lagra fyra kategorier av data:

Tokens

Fortnox Access Token och Refresh Token. Krypterade via Key Vault-referens. Inkluderar backup-kopia för tokenrotation.

ID-mappningar

Kopplingar mellan D365 GUID och Fortnox-nummer för kunder, produkter, offerter och ordrar.

Poll-tidsstämplar

Tidpunkt för senaste lyckade poll per resurstyp. Används som lastmodified-filter vid nästa poll.

Felkö

Misslyckade synkroniseringsförsök med payload och antal försök. Återförsöks automatiskt med exponentiell backoff.

Kapitel 5 — Datamappning

5.6 Betalningsstatus (Fortnox → D365)

När Fortnox rapporterar att en faktura är betald (Balance == 0) uppdaterar FortBridge motsvarande order i D365:

  • lg_invoicepaid sätts till true
  • lg_invoicepaiddate sätts till betalningsdatumet från Fortnox
  • D365-orderns statustransition till Fulfilled sker via Dataverse-actionen FulfillSalesOrder, inte via direkt PATCH på statecode
Konflikthantering: Om samma fält ändras i båda systemen parallellt vinner den senast mottagna ändringen. FortBridge använder ETag (If-Match) vid PATCH-anrop mot Dataverse för optimistisk concurrency, och loggar alla konflikter till Application Insights för uppföljning.

Kapitel 6 — Autentisering och säkerhet

Kapitel 6Autentisering och säkerhet

FortBridge hanterar autentisering mot tre externa system: Fortnox API, Dynamics 365 Dataverse och Azure-tjänster. Alla hemligheter lagras i Azure Key Vault och nås via Managed Identity — inga lösenord i konfigurationsfiler eller kod.

6.1 Nyckelhantering — hierarki

6.2 Fortnox-autentisering

Fortnox OAuth-flödet körs en gång vid installation. Därefter sköts tokenförnyelse automatiskt.

  1. Registrera integration på developer.fortnox.se. Notera Client ID och Client Secret.
  2. Kör installationsflödet: FortBridge erbjuder engångssidan /api/setup/authorize som startar OAuth-flödet och lagrar tokens krypterat i Key Vault.
  3. Automatisk förnyelse: RefreshFortnoxToken körs var 45 minuter, förnyar Access Token, och sparar det nya token-paret atomärt i Table Storage.

8.4 Idempotens — säker hantering av dubbla händelser

Alla FortBridge-synkroniseringsoperationer är idempotenta: att köra samma operation flera gånger ger samma resultat som att köra den en gång. Detta är kritiskt eftersom webhooks och polling kan leverera samma händelse fler än en gång.

Idempotens uppnås genom att:

  • Alltid söka på matchningsnyckeln (t.ex. lg_fortnoxcustomernumber) innan skapa — uppdatera om post finns, skapa annars
  • Jämföra checksummor på relevanta fält och hoppa över om inget har ändrats
  • Lagra senaste behandlade händelse-ID per entitet i Table Storage
  • Använda ETag (If-Match) vid PATCH-anrop mot Dataverse för att fånga upp parallella skrivkonflikter

8.5 Initial datamigration

Vid driftsättning på en befintlig miljö med existerande data i båda systemen behövs en engångsmigration för att matcha poster:

  1. Exportera alla Fortnox-kunder till CSV. Matcha mot D365-konton via organisationsnummer eller namn.

  4. Verifiera matchningskvalitet i Application Insights-loggar. Åtgärda omatchade poster manuellt innan FortBridge aktiveras för live-trafik.
Rekommendation: Kör initialmigration i en D365-sandbox-miljö först och verifiera resultatet noggrant. Behåll en fullständig export av D365-data som backup innan produktionsmigration.

Kapitel 9 — Realtidsprestanda

Kapitel 9Realtidsprestanda

Termen "realtid" är missvisande för en integration som måste passera två moln-API:er, serialisering och nätverk. Det praktiska målet är near-real-time: ändringar i ett system syns i det andra inom loppet av sekunder, inte minuter. Det här kapitlet beskriver hur FortBridge kan skalas från standard-konfigurationens 1–5 min latens till under 2 sekunder P95.

9.1 Latensbudget

Kapitel 9 — Realtidsprestanda

9.2 Tekniker för att minska latens

Webhooks över polling

Den enskilt största vinsten. Fortnox stöder webhooks för customer-*, invoice-paid och order-created. Dataverse Service Endpoints triggas vid Create/Update. Polling används bara som fallback för resurser utan webhook-stöd.

Premium Function Plan

Tar bort kallstarter helt och ger minst en alltid-varm instans. Kostnad ~2 200 kr/mån (EP1) men P95-latensen tappar ~1 500 ms. Rekommenderas när near-real-time är affärskritiskt.

HTTP Keep-Alive & connection pooling

Varm TLS-session sparar 150–300 ms per anrop. FortBridge håller en singleton-instans av HttpClient per målsystem under funktionsinstansens livstid.

Event Grid som buffert

För hög belastning: mottagande webhook-funktion postar direkt till Event Grid och svarar 200. Separata bearbetnings­funktioner konsumerar topicen. Kopplar bort inleveransen från bearbetningen och jämnar ut toppar.

Cacha ID-mappningar

Uppslag av D365-GUID ↔ Fortnox-kundnummer görs via minneskache (TTL 5 min) innan Table Storage-lookup. Träffar en varm cache tar < 1 ms istället för 30–80 ms.

Parallellisering med backpressure

Flera webhooks kan bearbetas samtidigt men med en semafor som begränsar samtidiga anrop mot Fortnox till 20 (hälften av rate limit). Förhindrar att egen last sänker oss till 429.

Kapitel 11Driftrunbok

Den här runboken beskriver åtgärder för de vanligaste incident­typerna i drift. Varje sektion listar symptom, första kontroll, och steg för återställning. Skriv alltid en kort incidentnotering i App Insights efter åtgärd.

11.1 Incidentklasser — översikt

KlassExempelÅtgärdsfönster
Kritisk (P1) Alla synkar stoppade, autentisering trasig < 1 timme, dygnet runt
Hög (P2) En entitetstyp misslyckas (t.ex. offerter) < 4 timmar, kontorstid
Låg (P3) Enskilda fel i dead letter queue Nästa arbetsdag

11.2 Refresh Token utgången

Symptom: Alla Fortnox-anrop returnerar HTTP 401. App Insights visar upprepade autentiseringsfel. Larm triggats.

Orsak: Refresh Token har inte använts på 60 dagar (integrationen stod stilla), eller roterades inte atomärt.

  1. Öppna /api/setup/authorize för berörd Fortnox-tenant
  2. Logga in som Fortnox-administratör och godkänn scopes på nytt
  3. Verifiera i Key Vault att nya tokens sparats (tidsstämpel < 5 min gammal)
  4. Kör HealthCheck med ?deep=true — båda anslutningar ska vara gröna
  5. Töm dead letter queue om poster finns — de kan nu behandlas

11.3 D365-certifikat utgånget

Symptom: Alla Dataverse-anrop returnerar HTTP 401. Felmeddelande innehåller "AADSTS700027: The certificate is invalid".

  1. Generera nytt certifikat (PowerShell New-SelfSignedCertificate eller CA) med minst 24 mån giltighet
  2. Exportera .cer (publik) och .pem (privat)
  3. Ladda upp .cer till app-registreringen i Azure AD
  4. Uppdatera PEM-secret i Key Vault — nytt versions-ID skapas automatiskt
  5. Starta om Function App så token-cachen rensas
  6. Verifiera anslutning via HealthCheck
Parallell giltighet: Ta inte bort det gamla certifikatet från app-registreringen förrän det nya är verifierat i drift. Azure AD stödjer flera aktiva certifikat — detta ger en säker rollback-möjlighet.

11.4 HTTP 429-storm från Fortnox

Symptom: Många HTTP 429-loggar, synkfördröjning ökar. Dead letter queue växer.

  1. Kontrollera i App Insights om en specifik timer-trigger fyrar för ofta — justera intervallet i host.json
  2. Kontrollera att Retry-After-header respekteras korrekt
  3. Om 429 persisterar: kontakta Fortnox support och be om höjd gräns (standard 400/min kan utökas mot avtal)
  4. Efter åtgärd: kör ReplayDeadLetterQueue manuellt

11.5 Dead letter queue växer

Symptom: Larm om >5 fel på 15 min. Poster ligger i Table Storage — failedSyncs.

  1. Öppna Azure Portal → Storage Account → Tables → failedSyncs
  2. Gruppera felposter på errorType — hitta gemensam nämnare

    • Kreditgräns nådd: Ekonomi höjer gränsen eller markerar kunden aktiv
    • Konflikt (412): ETag-mismatch — hämta nytt ETag och försök igen
  4. Kör ReplayDeadLetterQueue för åtgärdade poster

Symptom: Function App svarar 503 på /api/health?deep=true.

  4. Om Azure-incident misstänks: kontrollera status.azure.com för regionstatus

Access Token Kortlivad nyckel (1 timme i Fortnox) som skickas med varje API-anrop för att bevisa att integrationen har rätt att nå datan.

Dead Letter Queue Felkö — där misslyckade synkar hamnar efter tre automatiska retry:er. FortBridge lagrar dessa i Azure Table Storage.

ETag Versionsmarkör som Dataverse skickar med varje entitet. Används vid PATCH (If-Match-header) för att upptäcka parallella skrivkonflikter (optimistic concurrency).
HMAC-SHA256 Kryptografisk signatur (Hash-based Message Authentication Code med SHA-256). Används av Fortnox för att signera webhook-anrop så att mottagaren kan verifiera att anropet verkligen kommer från Fortnox.
Idempotens Egenskap där upprepad körning ger samma resultat som en körning. Kritiskt för webhook-hantering eftersom samma händelse kan levereras flera gånger.
Key Vault Azure-tjänst för säker lagring av hemligheter (tokens, certifikat, lösenord). FortBridge läser aldrig hemligheter från konfigurationsfiler — allt går via Key Vault.
Managed Identity En automatiskt genererad identitet som Function App använder för att autentisera mot Key Vault och andra Azure-tjänster — utan credentials i koden.
OAuth 2.0 Standardprotokoll för delegerad auktorisering. Fortnox använder Authorization Code Flow; FortBridge kör detta en gång vid installation.
OData Open Data Protocol — RESTful standard för datatillgång. Dataverse Web API är OData v4, vilket ger filter, expand och batch som standard.

TermFörklaring
PEM Privacy Enhanced Mail — textformat för certifikat och privata nycklar. Används för D365-autentisering (Application User med certifikat).
Polling Regelbunden avfrågning av ett API för att hitta ändringar. FortBridge pollar Fortnox var 5:e minut med ?lastmodified=-filter som komplement till webhooks.
Rate Limit Anropsbegränsning. Fortnox tillåter 400 anrop/min per integration. Vid överskridning: HTTP 429 med Retry-After-header.
Refresh Token Långlivad nyckel (60 dagar i Fortnox) som används för att hämta en ny Access Token när den gamla löpt ut. Roteras vid varje förnyelse.
Retry-After HTTP-header som talar om hur länge klienten bör vänta innan nästa försök. FortBridge respekterar alltid detta värde framför egen backoff-logik.
TermFörklaring
Scope (OAuth) Behörighetsomfång. Fortnox-scopes som customer, invoice, order styr vilka API-resurser integrationen får nå.
Service Endpoint (Dataverse) Mekanism i Plugin Registration Tool för att registrera webhook-mottagare. Ersätter tidigare "Plugin Steps" för extern webhook-integration.
Table Storage NoSQL-del av Azure Storage Account. FortBridge använder det för ID-mappningar, poll-tidsstämplar, och felkön.
Timer Trigger Azure Function-trigger som körs på schema (CRON-uttryck). FortBridge använder timers för token-förnyelse och polling.
UOM Unit of Measure — måttenhet. I D365 definieras UOM:er i enhetsgrupper (uomschedule). FortBridge måste alltid sätta uomid på orderrader, annars misslyckas skapandet.
Webhook Metod där källsystemet skickar en HTTP-request till mottagaren vid ändringar — realtidsalternativ till polling. Används från båda hållen i FortBridge.
