Kapitel 1 — Projektöversikt

Kapitel 1Projektöversikt & Beslut

NEXT är ett fullständigt byggprojekthanteringssystem som täcker hela projektets livscykel — från planering och upphandling till ekonomistyrning och fältarbete. Vi bygger full paritet (och mer) på Microsoft SharePoint Online med SPFX-webbdelar som primär leveransmekanism.

Vad vi bygger

Systemet levereras som ett SharePoint-tillägg med SPFX-webbdelar, Azure Functions för tung bearbetning och Power Automate för affärsflöden. Alla webbdelar byggs responsivt — mobilupplevelsen är inbyggd i SPFX, ingen separat app behövs.

Full paritet med NEXT

Alla moduler i NEXT replikeras eller förbättras. Ingen nedprioritering av funktionalitet — prioriteringslistan bestämmer ordning, inte omfattning.

SharePoint som plattform

Single-tenant SharePoint Online. Alla kärndata lagras i SharePoint-listor. Ingen extern databas — enkel backup, native sökning, inget underhåll.

Mobile-first SPFX

Webbdelar byggs responsivt från start. Fältarbetare använder SharePoint på mobil — inga appar eller extra licenser behövs.

       ╱             ╲
      ╱   Unit (300+) ╲ Jest · isolerat
     ╱                 ╲ ~15 s
    ─────────────────────

CI-gate: Unit+Integration obligatoriska per PR. E2E körs på merge till main (dev-env) och nightly (staging).

Coverage-mål & CI-thresholds
  • Unit: ≥60% line, ≥55% branch (hårdgate)
  • Integration: alla AF-endpoints täckta
  • E2E: kritiska flöden per roll (UAT-baseline)

  • Visual regression: Playwright screenshot-diff för 8 nyckelvyer

Bilaga AArchitecture Decision Records

De viktigaste arkitekturbesluten bakom NEXT, dokumenterade i kompakt ADR-format (Michael Nygard-stil). Varje post innehåller Kontext (problemet och begränsningarna), Beslut (vägen vi valde) och Konsekvenser (både positiva och negativa uppföljningar). Besluten är låsta för Fas 1 om inget annat anges — omförhandling kräver styrgruppsbeslut.

ADR-001 · SharePoint + SPFx som primär plattform

Kontext
Kundens M365-tenant är etablerad; användarna är redan inloggade. Alternativ: fristående Teams-app, extern SaaS (t.ex. Procore), eller custom React-app på Azure.
Beslut
SharePoint Online som datalager (listor, bibliotek) och SPFx som UI-lager. Ingen extern databas — all kärndata lagras i SP-listor.
Konsekvenser
+ Single sign-on gratis via Entra ID · native sök · versionshantering på biblioteken · backup/restore via M365 · ingen separat licens-kostnad. − SP list-throttling kräver batch-hantering · cross-site FK saknar integritet · uppgraderingstakt styrs av Microsofts SPFx-roadmap (1.22 idag, 2.x framöver) · mindre flexibilitet för exotiska UI-mönster.

ADR-002 · Azure Functions för tung bearbetning

Kontext
Vissa operationer (DWG→PDF-konvertering, OCR, BI-aggregering, Fortnox-sync) är för långsamma, för minnestunga eller kräver system-binärer (ODA) som inte kan köras i SPFx/PA.
Beslut
Separata Azure Functions per use-case, deployat i två planer: Consumption för event-drivna operationer (FortnoxSync, NotificationEngine, HealthCheck) och Premium EP1 för always-on latens-känsliga (DwgConverter, InvoiceOcrParser). Auth via Managed Identity.
Konsekvenser
+ Skalar automatiskt · ingen serverförvaltning · credentials hanteras utan secrets i kod · logs integrerade med Application Insights. − Cold start på Consumption (1–3 s) påverkar första anropet · Premium EP1 kostar ~1 900 kr/mån minimum · distribuerad felsökning kräver App Insights-disciplin.
