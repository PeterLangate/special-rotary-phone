Kapitel 1 — Syfte

Kapitel 1Syfte och bakgrund

FreeCorner är en intern SharePoint-lösning för Cornerstone som digitaliserar hela flödet från att en medarbetare ansöker om semester eller annan ledighet, via chefens attestering i Microsoft 365, till att den godkända ledigheten automatiskt registreras i löne­systemet Kleer (tidigare PE Accounting). Målet är att eliminera det manuella steget där medarbetare idag måste komma ihåg att själva registrera godkänd ledighet i Kleer — ett steg som regelbundet missas och leder till att semestern inte räknas av och vanlig lön betalas ut.

1.1 Problemformulering

I dagsläget godkänns ansökningar oftast manuellt via mail eller chatt. Chefen godkänner, men när månaden är slut glömmer medarbetaren att mata in ledigheten i Kleer. Resultat: vanlig lön betalas ut och semester räknas inte av. Administrativ rättning i efterhand är tidskrävande och felbenägen.

1.2 Mål

  • Ett enkelt formulär där medarbetare ansöker om ledighet (vem, när, betald/obetald, övrig info).
  • Automatisk routning till medarbetarens chef enligt Microsoft Entra ID.
  • Chef attesterar direkt i Teams eller via länk i mail — inget manuellt steg kvar.
  • Vid godkännande skickas ledigheten per dag till Kleer via deras API (X-Token).
  • Arbetsdagsberäkning — helger och röda dagar räknas bort automatiskt så att "5 dagars ledighet över en helg" aldrig kostar fem betalda semesterdagar.
  • Medarbetaren ser i ansökningsvyn sitt aktuella saldo: "Du har 3 tillgängliga semesterdagar".

1.3 Avgränsningar

Inom scope (v1)

Semester (betald/obetald), föräldraledighet, övrig ledighet med fri text, chefsattest, Kleer-synk, saldovisning, arbetsdagsberäkning.

Utanför scope (v1)

Sjukanmälan, VAB-registrering, chefsöverblick över teamets samlade frånvaro, automatisk vikarie­bokning, semesterplanerings­verktyg.

Framtida utbyggnad

Flerstegs­attest för långa perioder, kalender­vy per avdelning, Teams-bot för notiser, integration mot Outlook/kalender, sjuk­anmälan via samma flöde.

Kapitel 2 — Arkitektur

Kapitel 2Arkitektur

Lösningen består av en SharePoint-baserad front end (webpackad TypeScript-bundle deployad via .wsp-lösning) och en Azure Function back end (TypeScript, @azure/functions v4). Mönstret är återanvänt från Templates Manager (front end) och Teamsgeneratorn (back end) — båda i produktion hos andra Kalmstrom-kunder.

2.1 Komponentöversikt

2.2 Dataflöde i korthet

  1. Medarbetare öppnar Apply.aspx — bundlen läser profil och saldo.
  2. Formuläret POSTar till Function App: SubmitApplication.
  3. Function skapar SP-listpost, slår upp chefen via Graph, signerar en HMAC-token och mejlar chefen en länk till Approve.aspx?token=….
  4. Chefen öppnar länken, klickar Godkänn; bundlen POSTar till ApproveApplication.

Kapitel 3 — Flöde

Kapitel 3Ansöknings- och attestflöde

Ett flöde har fem status­steg: Utkast → Skickad → Godkänd eller Avslagen → Synkad. Varje övergång utlöser antingen ett användar­mail, en Kleer-anropskedja eller båda. Avvisade ansökningar går inte till Kleer.

3.1 Formulär (medarbetare)

FältTypBeskrivning
VemFörvaltAktuell användare, upplåst endast för HR
StartdatumDatumObligatoriskt, minst idag
SlutdatumDatumObligatoriskt, ≥ startdatum

3.2 Arbetsdagsberäkning

När användaren väljer 2026-06-05 till 2026-06-09 (måndag–fredag med Sveriges nationaldag mitt i) räknar systemet bara 4 arbetsdagar, inte 5. Logiken lever i WorkdayCalculator.ts och använder i Fas 1 en hårdkodad kalender som senare byts ut mot sholiday.faboul.se/dagar/v2.1/<år> (Fas 4).

Obs: Helger (lör/sön) räknas alltid bort. Röda dagar räknas bort för betald semester men inte för obetald ledighet — där räknas hela perioden i dagar per Kleers regel. Denna distinktion ligger i WorkdayCalculator.countPaidDays() vs countTotalDays().

3.3 Chefsattest

  1. Function slår upp chefen via Graph GET /users/{upn}/manager.
  2. Om chef saknas i Entra: ansökan stoppas med tydligt felmeddelande till medarbetaren — IT kontaktas.
  3. Mail till chefen med länk: https://<tenant>.sharepoint.com/sites/…/Approve.aspx?token=<hmac>.
  4. Token har 14 dagars TTL. Kort mejlpåminnelse (Fas 4) efter 7 dagar.
  5. Chef klickar Godkänn/Avslå. Kommentar är obligatorisk vid avslag.

Kapitel 5 — Kleer

Kapitel 5Integration mot Kleer

Kleer (f.d. PE Accounting) är Cornerstones ekonomi- och lönesystem. Integrationen sker från Function App till Kleers öppna API med X-Token-autentisering. Token lagras i Key Vault; ingen klient­sida ser token. Bas-URL: https://my.accounting.pe/api/v1/.

5.1 Anrop som används

MetodSökvägAnvänds för
GET/user/{id}Läsa saldo (vacation-*-days)
POST/event/Registrera en dags frånvaro
POST/event/approveMarkera registrerad frånvaro som attestad

5.2 Åtkomst och säkerhet

  • Nyckel-provisionering: Kleer lämnar ut API-nyckel + testbolag per avtal. Cornerstones IT begär in denna före Fas 3.
  • Förvaring: Token lagras i Azure Key Vault, hämtas av Function App via managed identity. Rotation var 12:e månad.
  • Nätverk: Bara Function App får prata med Kleer — klient­sidan går aldrig direkt dit.
  • Loggning: Varje anrop loggas till Application Insights med request-id men utan personnummer eller lönebelopp.

5.3 Felhantering och rekoncilering

Direkta fel

401/403 → larm till IT, token trasig. 4xx övrigt → KleerError sätts på posten, HR ser i admin-vy. 5xx → automatisk retry med exponentiell backoff (kRetryCore från shared).

Nattlig sweep

Timer-trigger i Function scannar poster där Status=Approved och KleerSyncedAt=null. Försöker igen. Rapporterar sammanfattning i veckovis mail till HR (Fas 4).

5.4 Mock-läge (Fas 1 och demo)

I Fas 1-mock:upen anropas Kleer inte alls. KleerClient.ts har en mockMode = true-flagga som i stället returnerar sekvensen "8 av 25 dagar tillgängliga" för saldo och en fördröjd success för event-POST. Detta räcker för demonstrationen för Björn och Robban utan att behöva avtal med Kleer.

Obs: När Kleer API-nyckeln är på plats, sätt KLEER_MOCK=false i Function App Configuration. Verifiera i testbolaget först — direktskrivning mot skarpt Kleer utan test kan förstöra lönedata.

Kapitel 6Leverans och drift

Lösningen levereras i fyra faser. Varje fas är självbärande — fas 1 kan demonstreras även utan beslut om fortsättning, och senare faser staplas på utan att riva det som redan är gjort.

FasInnehållStatus
0. HygienWorkspace, CLAUDE.md, brand-skill, denna systembeskrivningKlar
1. Mock:upSP-lösning, ansöknings­formulär, arbetsdags­math, fake-chef & fake-Kleer — för demo till Björn & RobbanPågår
2. AttestAzure Function, Graph manager-lookup, HMAC-mail­länk, Approve-sidan liveVäntar beslut

4. PolishRiktiga röda dagar, mailmallar, Playwright-smoke, admin-vyOption

6.2 Drift

SharePoint

.wsp deployas till Cornerstones SP Online-tenant. Uppdateringar via omdeploy av bundlen till siteassets/Ledighet.js.

Azure Function

Förvaltas i Cornerstones Azure-prenumeration. GitHub Actions eller VS Code-deploy. Managed identity mot Key Vault och Graph.

Övervakning

Application Insights för Function. SP-listans Views ger HR direkt överblick över synk­status utan separat dashboard.

6.3 Beroenden och oklara punkter

  • Kleer API-nyckel — krävs före Fas 3. Begäran skickas av IT till Kleers support.
  • Graph-behörighet — Function App behöver User.Read.All (applikations­behörighet) för att läsa chef för godtycklig användare. Kräver global admin-samtycke.
  • HMAC-hemlighet — genereras vid första deploy och läggs i Key Vault. Rotation vid behov.
  • Delarbetstid — Fas 1–3 antar heltid (8 h/dag). Deltidsstöd kräver att Kleer-user har korrekt kalender, kontrolleras i Fas 4.

Kapitel 8Mailnotiser

FreeCorner driver hela flödet via mailnotiser med signerade länkar. Chefen behöver inte logga in i SharePoint för att godkänna eller avslå — länken i mailet räcker. Detta är medvetet: minsta möjliga friktion är nyckeln till att ansökningar inte blir liggande.

Kapitel 8 — Mailnotiser

8.6 Säkerhet kring mail-länkar

HMAC-signerad token

Varje länk till Approve.aspx bär en HMAC-SHA256-signerad token. Secret-nyckeln lever i Azure Key Vault och roteras årligen. Tokenet kapslar in SP-ID, applicant-UPN, manager-UPN, utfärdandetid och beslutstyp. Ingen användarinput skickas i URL:en — bara signerade fält.

14 dagars TTL

Function verifierar utfärdandetiden. En länk äldre än 14 dagar avvisas med tydlig felsida och uppmanar chefen att öppna ärendet direkt i appen. Ingen återanvändning — efter ett beslut kastas token i en blacklist-tabell.

Mottagarkontroll

När chefen klickar länken kontrollerar Function även att den inloggade användaren (SP-kontext) matchar managerUpn i token. Skickas länken vidare till någon annan ger det en “Inte din ansökan”-sida — inte en tyst godkännande.

Ingen känslig data i URL

Personnummer, lönebelopp och övrig info är inte del av tokenet. URL:en avslöjar bara id. Själva ansökningsinnehållet hämtas av Approve.aspx via autentiserad SP-REST när sidan laddas.

8.7 Mallförvaltning

Mailmallarna ligger i LedighetFn/src/templates/*.html (Fas 2) och är HTML-mail med inline-CSS enligt e-post-best-practice (tabellbaserad layout för Outlook, inline-stilar, max 600 px bred). Varje mall har en svensk och en engelsk version — språk väljs utifrån mottagarens preferredLanguage i Entra. Fallback är svenska.

8.8 Distribution och revisionsspår

HR-mailet (mail 4)

Går till distributionslistan hr@cornerstone.se — inte till enskild HR-partner per avdelning. Beslut lockat 2026-04-24. Om HR-teamet växer eller delas per ort kan det brytas ut senare genom att mappa applicantens department-attribut till olika listor, men det är inte nödvändigt nu.

Inget BCC på beslutsmail

Mail 2 och 3 (godkänt / avslag till medarbetaren) BCC:ar inte HR. SharePoint-listan Ledighetsansökningar är revisionsspåret — HR kan alltid se Status, ManagerComment, DecidedAt, ApprovalToken per post. Ingen anledning att fylla HR:s inkorg med kopior. Beslut lockat 2026-04-24.

Obs vid utrullning: Mail kan ta upp till 3 minuter att nå inkorgen (Graph-kö + Exchange). Informera cheferna i introduktionsmailet så ingen undrar varför länken inte kom direkt. HR-synkfel-mail har högre prioritet och går via en direkt SMTP-relay (Fas 3).

Kapitel 9Varianter

Samma grundfunktion — ansökan, chefsattest, revisionsspår — kan paketeras i fyra varianter beroende på UI (formulär eller AI-chatt) och integrationsdjup (automatisk Kleer-synk eller manuell). Alla fyra delar backend (Azure Function + SharePoint-listan). UI-lager och integrations­djup är de enda axlarna som varierar.

9.3 Variant B — utan lönesystem-synk

Identisk med A, förutom att Synkstatus-fliken och saldokortet inte finns. Mina ansökningar har en extra kolumn Registrerad i lön — en bockruta medarbetaren själv markerar efter manuell inmatning. Mail 2 har ett tydligt "Kom ihåg att registrera i lönesystemet"-block.

9.4 Variant C — AI-chatt, lätt

En enda roll (medarbetaren) pratar med assistenten. Samma ansöknings- och attestflöde bakom, men utan automatisk Kleer-synk. Bra för kunder som vill testa AI-gränssnittet innan man lyfter Kleer-integrationen. Baserad på mönstret från SPAI-projektet.

9.5 Variant D — Full AI (AI + Kleer)

Alla tre roller pratar med AI:n. Medarbetaren ansöker, chefen godkänner, HR följer upp synkstatus — allt via samma chatt. Kleer är djupt integrerat: saldo läses live när användaren frågar, event-POST mot Kleer sker i realtid, och synkfel visas som strukturerade kort direkt i HR-chatten.

Medarbetare-chatten

"Vad har jag för saldo?" → AI:n läser live från Kleer och svarar med tillgängliga / använda / sparade dagar. "Ansök 6–31 juli" → AI:n räknar arbetsdagar, verifierar att saldot täcker, skapar SP-post och mailar Per.

Chef-chatten

"Jag har 3 ansökningar till dig idag." → Per kan be AI:n visa hela teamets saldon (Kleer-data för 8 personer på en gång), godkänna en efter en, och se direkt att godkännandet har skrivit 20 events till Kleer — utan att lämna chatten.

HR-chatten

AI:n rapporterar proaktivt synkfel: "47 försök idag, 46 ok, 1 fel." Lisa kan be om detaljer och instruera AI:n att försöka igen eller skicka ett mail med instruktion till chefen för den misslyckade posten.

Gemensam bas

Samma ChatOrchestrator Function, samma systemprompt, men tool-kedjan anpassas efter inloggad roll (Entra group membership avgör vilka verktyg AI:n får anropa). Inget extra kod-lager per roll.

Grönt “Från Kleer (live)”-kort signalerar att värdena kommer direkt från lönesystemet — inte från cache eller användarminne.

9.7 Prissättning och leveransordning

  • Fas 1–3 levererar Variant B (utan synk) som MVP.
  • Fas 3-tillägg aktiverar Kleer-synk → Variant A.
  • Fas 5 (option) lägger till chatt-lagret — kan levereras som C ovanpå B (om Kleer inte är aktivt) eller D ovanpå A (om Kleer är aktivt). Priset styrs av vilken du stackar på.
  • Variant D kräver dessutom Entra-grupper konfigurerade så AI:n vet vem som är chef respektive HR innan den väljer verktygsset.
AI-revisionsspår: I både C och D loggar Function exakt vad AI:n “bestämde sig för” per meddelande (prompt + response + verktyg-kedja + beslut). Revisionsspåret i SP-listan inkluderar alltså inte bara vem som skickade in ansökan utan hur — viktigt för regulatoriska krav och för att spåra AI-fel.
