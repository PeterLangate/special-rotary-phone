Chapter 1 — Introduction

Chapter 1Introduction

The Mermaid Webpart brings professional, text-based diagramming to SharePoint classic pages. Before exploring the webpart itself, this chapter explains what Mermaid is, why text-based diagrams outperform drawing tools for most organisational use cases, and what the webpart adds on top of the Mermaid engine.

1.1 What is Mermaid?

Mermaid is an open-source JavaScript library that converts plain text into SVG diagrams directly in the browser. It was created in 2014 by Knut Sveidqvist and has since grown into one of the most widely adopted diagramming systems in the software industry. Major platforms — GitHub, GitLab, Confluence, Notion, and Azure DevOps — all render Mermaid natively in their Markdown editors.

The core idea is simple: instead of drawing a diagram by dragging boxes and arrows, you describe the diagram in a concise notation, and Mermaid renders it. A flowchart that would take ten minutes to arrange by hand can be written in seconds:

graph TD
    A[Customer Request] --> B{In Stock?}
    B -->|Yes| C[Process Order]
    B -->|No| D[Back Order]

This notation is instantly readable by a human, can be stored in any text file, and produces a clean, consistently styled vector diagram every time it is rendered.

1.2 Why text-based diagrams?

Most organisations still use visual drawing tools — Microsoft Visio, PowerPoint SmartArt, Lucidchart, or browser-based alternatives — to create diagrams. These tools have well-known limitations that text-based approaches solve directly.

Consistent style, automatically

In drawing tools, every person who edits a diagram can change fonts, colours, box sizes, and spacing. Mermaid applies a consistent visual theme to every diagram, every time, with no manual alignment effort.

Easy to update

Adding a new step to a Visio flowchart requires moving every downstream box to make room. In Mermaid you add one line of text and the layout recalculates automatically.

Version control friendly

Because Mermaid diagrams are plain text, they can be stored in source control, diffed line by line, and reviewed in pull requests — impossible with binary drawing files.

No software licence required

Mermaid runs in any browser, free of charge. Viewers do not need Visio, Lucidchart, or any other installed tool to see a rendered diagram.

Lives where the content lives

A Mermaid diagram is stored as text alongside the page it illustrates. It does not require export, file attachment, or a separate link to an external system.

Accessible output

Mermaid renders SVG, which is natively accessible. Accessibility titles and descriptions can be added directly in the code with accTitle and accDescr directives.

1.3 When NOT to use Mermaid

Mermaid is excellent for structured, logic-driven diagrams. It is a less natural fit when:

  • Precise visual layout matters. Mermaid controls the layout algorithmically. If you need exact pixel placement — floor plans, hardware schematics, custom infographics — a drawing tool is more appropriate.
  • Non-technical authors create diagrams. Mermaid requires learning a small syntax. The built-in examples and snippets reduce this barrier significantly, but some users will still prefer a visual tool.
  • Diagrams change constantly in real time. If the underlying data changes daily (e.g. a live org chart), a data-connected diagramming solution or auto-generated chart is more maintainable.

1.4 The Mermaid ecosystem

Because Mermaid is open-source and widely adopted, a large ecosystem has grown around it:

1.5 What the Mermaid Webpart adds

The Mermaid engine is a JavaScript library. Integrating it into a SharePoint classic webpart requires solving several additional problems that the raw library does not address:

  • Persistence. The Mermaid code must be saved and reloaded with the page. The webpart stores code in the SharePoint webpart property bag, so diagrams survive page saves, deployments, and browser refreshes.
  • Edit workflow. Classic SharePoint has no native code editor. The webpart provides a full-screen modal dialog with code editor, live preview, and save/cancel controls.
  • Discovery. New users do not know Mermaid syntax. The built-in example library (40+ ready-made diagrams) and snippet menu let users produce useful diagrams immediately without reading the Mermaid documentation.
  • Click handler authoring. The Mermaid click syntax is not obvious to non-developers. The Add Click Handler tool generates correct click syntax from a form, making interactive diagrams accessible to all authors.
  • Multi-instance support. Multiple diagrams can exist on the same page. Each webpart instance maintains its own code independently.

Chapter 2 — Getting started

Chapter 2Getting started

This chapter explains how to add the Mermaid Webpart to a SharePoint page and what you see when it first loads.

2.1 Adding the webpart to a page

The webpart is deployed to SharePoint by an administrator and appears in the classic webpart gallery. To add it to a page:

  1. Open the target SharePoint page in edit mode (click Edit in the page ribbon or Site Actions > Edit Page).
  2. Click inside the webpart zone where you want the diagram to appear.
  3. From the Insert tab in the ribbon, choose Web Part.
  4. In the gallery, find Mermaid Webpart under the appropriate category and click Add.
  5. The webpart is inserted and shows a default diagram immediately.
  6. Save the page. The diagram is now visible to all visitors.
Tip: You can add more than one Mermaid Webpart to the same page. Each instance stores its own diagram code independently.

2.2 The default diagram

When the webpart is first placed on a page, it displays a built-in example that demonstrates several Mermaid features in one diagram: coloured nodes, styled links, and clickable nodes that open URLs in a new tab.

The default diagram is a site architecture chart showing:

  • A Home node linking to Projects and Admin Center
  • Two project leaf nodes (Project 1, Project 2)
  • Custom colours applied via classDef declarations
  • Styled arrows (solid and dashed) via linkStyle
  • Clickable nodes using Mermaid click syntax

You can replace this diagram at any time by opening the code editor (see Chapter 3).

2.3 How diagrams are saved

The Mermaid code for each webpart instance is saved in the SharePoint page itself using the standard webpart property bag. No external lists or databases are required. The diagram travels with the page and is immediately available after saving.

For administrators: The webpart uses the SharePoint webpart storage mechanism. Backup and restore of diagrams follows normal page backup procedures. There is no separate data store to manage.

2.4 Editing an existing diagram

To change a diagram that is already on the page:

2.6 Mobile and tablet use

The webpart behaves differently in view mode compared to edit mode on small screens:

  • View mode: The rendered SVG diagram scales to fill the full width of the webpart zone. It is readable on any screen size. Interactive (clickable) nodes respond to taps on touch devices.
  • Edit mode: The code editor is a fixed-size dialog (900 × 600 px). On small screens or with an on-screen keyboard active, the dialog may be partially obscured. Authoring diagrams is best done on a desktop browser with a physical keyboard.

Chapter 3 — The code editor

Chapter 3The code editor

The Mermaid Code Editor is the main dialog for writing and previewing diagram code. It opens as a modal window (900 × 600 px, resizable) with the code panel on the left and a live preview on the right.

3.1 Editor layout

Code panel (left)

A text editor pre-loaded with the current diagram code. Supports Tab for indentation (inserts two spaces). Spell-check is disabled so Mermaid keywords are not highlighted as errors.

Live preview (right)

Automatically re-renders the diagram 500 ms after you stop typing. If the code contains a syntax error the preview area shows the error message in red instead of a diagram.

A splitter bar between the two panels can be dragged left or right. Each panel has a minimum width of 250 px.

3.2 Writing Mermaid code

Mermaid code starts with a diagram type declaration, followed by the content specific to that type. For example, a basic flowchart:

Each diagram type has its own syntax. The built-in examples (Chapter 5) demonstrate the most common patterns. For a full language reference visit the official Mermaid documentation at mermaid.js.org.

3.3 Live preview behaviour

The preview updates automatically while you type. There is a 500 ms debounce delay, so the engine waits until you pause before re-rendering. This prevents unnecessary re-renders on every keystroke.

When the code panel is empty, the preview displays the placeholder text "Enter Mermaid code to see preview…"

When the code contains a syntax error, the preview shows a red error box with the engine's error message. The error does not affect the saved diagram — the previous valid diagram remains on the page until you save a new valid one.

3.5 Saving and discarding changes

The toolbar at the bottom of the editor contains two controls:

  • OK — closes the dialog, saves the new Mermaid code, and re-renders the diagram on the page. The page still needs to be saved via the SharePoint ribbon to persist the change.
  • Closing the dialog (window × button) — discards all changes made in this editor session. The diagram on the page is unchanged.
Remember: Clicking OK saves the code to the webpart but does not save the SharePoint page. Always save the page afterwards using the ribbon or page controls, otherwise the change is lost on the next page load.

3.6 Version and instance information

A grey information row at the bottom of the editor shows:

  • The webpart version number (e.g. Mermaid Webpart 0.1.4.5).
  • The unique instance ID of the webpart on the right. Hovering over this shows the tooltip "This is the unique ID of this webpart". This ID is useful when reporting issues to an administrator.

Chapter 4 — Tools

Chapter 4Tools

The editor toolbar contains three tools that help you write Mermaid code faster and add interactivity to diagrams. A right-click context menu on the preview panel provides additional node-level actions.

4.1 Edit in Live

The ✏️ Edit in Live link opens the current diagram code in the Mermaid Live Editor in a new browser tab. The Mermaid Live Editor is the official online tool provided by the Mermaid project and offers additional features such as:

  • Diagram configuration panels
  • PNG and SVG export
  • Shareable diagram URLs
  • Alternative rendering engines

After editing in the Live Editor, copy the code from there and paste it back into the webpart editor. Changes made in the Live Editor are not automatically synchronised back to SharePoint.

4.2 Snippets

The 📋 Snippets button opens a categorised drop-down menu of reusable code fragments. Clicking a snippet inserts it at the current cursor position in the editor.

Chapter 6 — Diagram type reference

Chapter 6Diagram type reference

Each Mermaid diagram type is declared by a keyword on the first line of the code block. This chapter describes all supported types, their starting keyword, a minimal code example, and their primary use case.

6.1 Flow and process diagrams

Flowchart

Keywords: graph TD  /  graph LR  /  flowchart TD

The most versatile diagram type. Nodes connected by arrows represent decisions, steps, and data flows. Direction is set after the keyword: TD (top-down), LR (left-right), TB (top-bottom), BT (bottom-top), RL (right-left). Supports subgraphs, custom node shapes, class-based styling, link styles, and clickable nodes.

graph TD
    A[Start] --> B{Ready?}
    B -->|Yes| C[Continue]
    B -->|No| D[Wait]

Sequence diagram

Keyword: sequenceDiagram

Shows the chronological order of messages between participants (people, services, systems). Supports actors, alternative blocks (alt/else), parallel execution (par), loops, and notes. Participants can have clickable links via the link directive.

sequenceDiagram
    Client->>Server: GET /data
    Server-->>Client: 200 OK

State diagram

Keyword: stateDiagram-v2

Models the states an object or system can be in and the transitions between them. Useful for documenting order lifecycles, UI screen flows, or any process with distinct states.

stateDiagram-v2
    [*] --> Pending
    Pending --> Shipped : confirm
    Shipped --> [*]

User journey

Keyword: journey

Plots a user experience across sections and steps, with a satisfaction score (1–5) and the actors involved at each step. Good for UX analysis and customer journey mapping.

journey
    title Checkout flow
    section Browse
        Search: 5: Customer
        Add to cart: 4: Customer

6.2 Data and relationship diagrams

Entity Relationship diagram

Keyword: erDiagram

Describes database tables (entities), their attributes, and the relationships and cardinalities between them. Uses standard ER notation: ||--o{ (one-to-many), }|..|{ (many-to-many), etc.

erDiagram
    ORDER ||--o{ ITEM : contains
    ORDER { int id string status }

Class diagram

Keyword: classDiagram

Documents object-oriented class structures, interfaces, inheritance hierarchies, and associations. Supports method signatures, access modifiers (+/-/#), and generics notation.

classDiagram
    Animal <|-- Dog
    Animal : +String name
    Animal : +speak()

Requirement diagram

Keyword: requirementDiagram

Traces system requirements to implementing components. Each requirement has an ID, description, risk level, and verification method. Components are linked to requirements they satisfy.

requirementDiagram
    requirement REQ_01 {
        id: 1
        text: "System must log all actions"
        risk: medium
    }

6.3 Architecture diagrams

Architecture diagram beta

Keyword: architecture-beta

Renders infrastructure and service architecture using named icons (server, database, internet, cloud). Services are grouped into subnets or regions and connected with directional connectors.

architecture-beta
    group cloud(cloud)[Cloud]
        service db(database)[Database] in cloud
        service api(server)[API] in cloud
    api:R --> L:db

C4 diagram

Keyword: C4Context / C4Container / C4Component

Implements the C4 model for software architecture documentation. C4Context shows system context (people, systems, boundaries). Relationships include communication descriptions.

C4Context
    Person(user, "User")
    System(app, "Web App")
    Rel(user, app, "Uses", "HTTPS")

Pie chart

Keyword: pie

Simple proportional chart. Each slice is defined by a label and a numeric value. A title can be provided after the keyword.

Keyword: xychart-beta

Renders bar and line series on a common X/Y axis. Axes are explicitly defined with labels and value ranges. Multiple series types can appear on one chart.

Four-quadrant scatter plot where each quadrant is named. Items are placed by X and Y coordinates (0.0–1.0). Good for priority matrices and BCG-style analysis.

Preview shows a red error box Syntax error in the Mermaid code. Read the error message — it usually names the line or token that failed. Check spelling of the diagram keyword, unclosed brackets, or an invalid arrow style. The previous valid diagram is preserved on the page.
Diagram disappeared after saving The SharePoint page was not saved after clicking OK. Clicking OK saves code to the webpart in memory, but the SharePoint page must also be saved via the ribbon (Save or Stop Editing). Reload the page to confirm the change persists.
Edit button / editor does not open The page is not in edit mode. Click Edit in the SharePoint ribbon or choose Site Actions › Edit Page first, then click the webpart edit handle and choose Edit Web Part.
Click handlers do not navigate Browser popup blocker is active, or the URL is malformed. Allow popups from the SharePoint site in the browser. Verify the URL in the click syntax is absolute (starts with https://). Relative URLs are not supported.
Node ID list in Add Click is empty The editor could not parse node IDs from the current code. The parser only recognises standard Mermaid node declarations (A[label], A(label), etc.). Ensure the code is valid and that at least one node is declared before opening the dialog.
Edit in Live opens a blank page Browser popup blocker prevented the new tab from opening. Allow popups from the SharePoint site, or right-click the Edit in Live button and choose Open link in new tab manually.
Old diagram still shows after redeployment Browser cache is serving the previous JavaScript bundle. Perform a hard refresh: Ctrl+Shift+R (Chrome/Edge) or Ctrl+F5 (Firefox). If the issue persists for all users, ask an administrator to clear the SharePoint blob cache.
Diagram is cut off or too small The webpart zone is narrower than the rendered SVG. Mermaid SVG scales to fill available width. Use a wider webpart zone, or reduce diagram complexity. For wide diagrams, switch to left-right layout: replace TD with LR in the diagram keyword.
Right-click context menu does not appear Right-click was on the background, not on a diagram node. Right-click directly on a coloured node shape. The context menu only appears on Mermaid nodes, not on the empty area around the diagram.

7.3 Known limitations

The following limitations apply to all versions of the Mermaid Webpart:

LimitationDetail
No layout control Mermaid positions all nodes algorithmically. You cannot drag nodes to custom positions. For diagrams requiring exact placement, use Visio or PowerPoint instead.
Static data only Diagrams are hand-authored text. They do not update automatically when underlying data changes. For live org charts or data-driven visuals, a different solution is needed.
No embedded images Mermaid does not support embedding raster images inside diagram nodes. Only text labels, Font Awesome icons, and shapes are available.
Large diagrams can be slow Diagrams with 50+ nodes may take a noticeable time to render. If the preview is slow, reduce the number of nodes or split the diagram into smaller pieces.
Changes lost without page save Clicking OK saves to webpart memory only. If the browser is closed or the session expires before the SharePoint page is saved, the change is lost.
Beta diagram types may change Types marked -beta (kanban, xychart-beta, sankey-beta, treemap-beta, architecture-beta) may have their syntax changed or removed in future Mermaid engine updates.

7.4 Mobile and tablet use

The Mermaid Webpart has two distinct behaviours depending on whether the page is in view mode or edit mode:

View mode (all devices)

The rendered SVG diagram scales to fill the full width of the webpart zone. It is responsive and readable on phones and tablets. Interactive nodes respond to taps. No editor is available in view mode regardless of device.

Edit mode (desktop recommended)

The code editor dialog is a fixed-size modal (900 × 600 px). On small screens it may overflow or be partially hidden behind the keyboard. Editing is best performed on a desktop browser with a physical keyboard.
